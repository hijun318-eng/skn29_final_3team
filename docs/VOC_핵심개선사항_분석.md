# 호텔 VOC·운영 이슈 분석 Agent 핵심 개선사항

## 1. 분석 목적

[추가 제안] 이 문서는 두 논문의 방법론을 현재 기획 중인 「호텔 VOC·운영 이슈 분석 Agent」와 비교하여, 단순 분류·탐지 기능을 실제 개선 실행과 재발 추적으로 연결할 기능을 도출한다.

[현업 검증 필요] 논문의 연구 대상은 A호텔과 관광 커뮤니티 비즈니스의 마을호텔이며 워커힐이 아니므로, 논문의 빈도·순위·품질 유형·개선 효과는 워커힐 현황으로 사용하지 않는다.

[현재 구현] 분석 기준은 사용자 제공 기능 목록과 1491×1055 해상도의 `호텔voc.png`에 표시된 기획안이며, 실제 운영 배포나 내부 데이터 보유를 증명하는 자료로 취급하지 않는다.

## 2. 현재 서비스 구현 범위

[현재 구현] 기획안은 VOC 감성·주제·긴급도 분류, 유사 사례 검색, 반복 불만 탐지, 운영 KPI 교차분석, 이동평균·Z-score 이상 탐지, 원인 후보, 권고안, 대시보드·원인 보고서·주간 보고서, 담당자 검토를 범위로 둔다.

[현재 구현] 이미지에는 리뷰/VOC, 시설 운영, 객실 예약, 인력·대기시간, 날씨·행사 데이터 후보와 PostgreSQL·pgvector·S3·LLM·FastAPI·Next.js 등의 기획 스택이 표시되어 있다.

[내부 데이터 필요] 위 데이터는 필요 데이터 후보일 뿐 워커힐의 실제 보유·연계 가능 데이터로 확인되지 않았으며, 원문·처리 이력·운영 KPI·SLA·조치 이력의 제공 여부를 별도로 확인해야 한다.

## 3. 논문별 적용 가능한 방법론

### 3.1 A호텔 VOC 사례연구

[논문 근거] 이지현의 2019년 석사학위논문은 A호텔 VOC 게시판의 2016~2017년 자료를 사례연구로 분석하고, 빈도 분석·대표 단어 코딩·유사 개념 범주화·반복적 비교 분석·SERVQUAL 분류를 사용했다. 근거는 PDF pp.49~63(본문 pp.33~47)이다.

[논문 근거] 수집 필드는 등록/발생 일시, 발생지, VOC 유형, 서비스 유형, 접수 경로, VOC 내용, 답변 내용, 처리 내용, 처리 부서, 답변·처리 담당자, 처리시간이다. 논문 개요의 `등록 일시`와 본문의 `발생 일시`, 수집표를 함께 확인했으며 근거는 PDF pp.14, 51~52(논문 개요, 본문 pp.35~36)이다.

[논문 근거] 서비스 범주는 운영·접객 서비스·시설·상품·청결·기타이며, 대표 단어를 주제별로 코딩한 뒤 유사 유형을 묶어 빈도를 계산했다. 근거는 PDF pp.53~55(본문 pp.37~39)이다.

[논문 근거] 분석은 칭찬·불만 분류, 발생 장소별·서비스 유형별 빈도, 대표 단어 코딩, 유사 유형 범주화, 감정표현·발화 특성·요구 내용·불만 원인 검토로 구성된다. 근거는 PDF pp.49, 53~63(본문 pp.33, 37~47)이다.

[논문 근거] SERVQUAL은 신뢰성·확신성·유형성·공감성·반응성으로 코딩했으며 한 VOC가 두 요소 이상을 포함하면 중복 코딩했다. 근거는 PDF pp.56~60(본문 pp.40~44)이다.

[논문 근거] 불만 내용은 감정표현·발화 특성·구체적 요구·불만 주체·상황·상대방 고려 등을 반복 비교했으며, 연구 결과로 누적형 불만·차별대우·직원 실수·주관적 기준·객관적 기준 유형을 제시했다. 근거는 PDF pp.61~63, 85(본문 pp.45~47, 69)이다.

[현업 검증 필요] 위 다섯 원인 유형은 A호텔 자료에서 연구자가 해석한 결과이므로 워커힐의 원인 코드로 확정하지 않고, 모델이 제시할 수 있는 검토 후보 코드로만 사용해야 한다.

[논문 근거] 논문은 칭찬·불만만 분석한 범위, 단일 A호텔의 일반화 한계, 익명 자료의 인구통계 분석 한계, SERVQUAL 판단의 주관성을 명시했다. 근거는 PDF pp.95~96(본문 pp.79~80)이다.

### 3.2 SERVQUAL-Kano-QFD 연구

[논문 근거] 지윤호·한장헌·한주형의 2025년 연구는 서비스 이용 전 기대 수준과 이용 후 지각 수준의 차이로 서비스품질을 평가하는 SERVQUAL 개념을 설명한다. 근거는 PDF p.5(학술지 p.131)이다.

[논문 근거] Kano는 매력적·일원적·당연적·무관심·역 품질로 구분하며, 한 속성에 대한 긍정 질문과 부정 질문의 쌍대 설문으로 평가한다. 근거는 PDF pp.6~7(학술지 pp.132~133)이다.

[논문 근거] QFD는 VOC의 고객 요구를 서비스·운영·기술 요구로 변환하고 서비스 설계·예약·운영·판매 단계의 개선 요구를 연결하는 방법이며, HOQ는 고객 요구와 개선 수단의 관계 및 상대적 중요도를 매트릭스로 표현한다. 근거는 PDF pp.3, 7~8(학술지 pp.129, 133~134)이다.

[논문 근거] 연구는 Kano 설문 결과에서 CSC·CDC를 산출하고 HOQ 관계 강도에 강 5점, 보통 3점, 약 1점을 부여했다. 근거는 PDF p.13(학술지 p.139)이다.

[논문 근거] 연구의 표본은 2023년 11월 11일~12월 10일 네 마을호텔 숙박객을 대상으로 한 대면 설문 128부이며, 첫 방문자가 73.4%였다. 근거는 PDF pp.10~11(학술지 pp.136~137)이다.

[현업 검증 필요] 객실 보안·물온도·어메니티·향기, 노스텔지어·사랑방·마을호텔 VOC 관리 등의 유형·순위는 해당 표본의 연구 결과이므로 워커힐 우선순위나 Kano 유형으로 전용하지 않는다.

## 5. 현재 서비스의 핵심 한계

1. [추가 제안] 분류 결과가 `무엇이 문제인가`에는 답하지만 고객이 기대한 상태, 실제 경험, 차이, 요구 조치를 분리하지 않아 개선 요구 정의가 약하다.
2. [추가 제안] 유사 검색과 반복 탐지가 일회성 결과로 끝나며, 클러스터의 병합·분리·대표 사례·담당 부서·조치 이력·재발 상태를 관리하지 않는다.
3. [추가 제안] 원인 후보와 권고안에 원문 근거, 운영 근거, 반대 근거, 확인할 사실, 규정 버전이 강제되지 않아 생성문 과신 위험이 있다.
4. [추가 제안] 우선순위의 입력 요인과 가중치 출처가 정형화되지 않아 모델의 임의 점수처럼 보일 수 있다.
5. [추가 제안] 고객 요구를 운영 과제로 바꾸는 관계표와 담당 역할·상태·기한을 가진 개선 백로그가 없다.
6. [추가 제안] 보고서가 조치 이후 동일 문제의 재발, 미해결, SLA, 담당자 수정률까지 폐쇄루프로 추적하지 않는다.
7. [현업 검증 필요] 안전·차별·직원 과실·보상·법적 책임·인과효과는 AI가 확정할 수 없지만, 각 항목별 필수 검토 역할과 승인 게이트가 더 구체적이어야 한다.
8. [내부 데이터 필요] 운영 KPI와 VOC의 시간·업장·서비스 접점 키, 노출량 분모, SLA, 조치 ID가 없으면 이상 탐지와 개선 효과 추적의 해석력이 제한된다.
9. [추가 제안] 감정표현 강도·긴급도와 불만 원인 후보는 서로 다른 필드로 저장해야 하며, 격한 표현을 특정 원인이나 고객 책임의 근거로 사용하지 않는다.

## 6. 추가해야 할 VOC 핵심 개선사항

### 개선사항 1. VOC 분류에서 서비스 품질 갭 분석으로 확장

- 논문 근거: [논문 근거] SERVQUAL은 이용 전 기대와 이용 후 지각의 차이를 서비스품질 분석의 핵심으로 본다(논문 2, PDF p.5).
- 현재 서비스 구현 내용: [현재 구현] 감성·주제·긴급도와 원인 후보를 분류한다.
- 현재 한계: [추가 제안] 고객의 주장 안에서 기대 상태와 실제 경험이 분리되지 않아 개선 대상과 이유가 모호하다.
- 추가 기능: [추가 제안] `expected_service`, `perceived_service`, `service_gap`, `customer_requested_action`, `fact_to_verify`, `evidence_sentence`, `reviewer_status`를 추출한다.
- 필요한 데이터: [내부 데이터 필요] 마스킹 VOC 원문, 예약·상품 설명·약관·현장 안내·처리 기록의 해당 시점 버전이 필요하다.
- 분석 방법: [추가 제안] 텍스트의 명시적 문장만 추출하고, 암묵적 기대는 `inferred=true`와 신뢰도를 표시하며 정책 근거와 대조한다.
- 사용자에게 제공되는 결과: [추가 제안] 개선 대상, 기대-경험 차이, 확인할 사실, 근거 문장, 관련 운영 지표를 한 카드로 제공한다.
- Human-in-the-loop: [현업 검증 필요] 담당자가 고객 주장·정책상 약속·확인된 사실을 구분하고 갭 문구를 승인한다.
- MVP 구현 가능 여부: [MVP 가정] 원문 기반 후보 추출은 가능하지만 객관적 SERVQUAL 점수로 표현하지 않는다.
- 기대 효과: [추가 제안] 분류 라벨을 실행 가능한 고객 요구 정의로 전환할 수 있는 기반을 만든다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] 고객의 서술만으로 실제 서비스 실패나 호텔 책임이 확인됐다고 단정하지 않는다.

### 개선사항 2. 단일 감성 분류에서 SERVQUAL 다중 라벨로 확장

- 논문 근거: [논문 근거] A호텔 연구는 한 VOC에 둘 이상의 SERVQUAL 요소가 있으면 중복 코딩했다(논문 1, PDF p.60).
- 현재 서비스 구현 내용: [현재 구현] 감성·주제 분류가 기획되어 있고 기존 기업분석 문서에는 SERVQUAL 후보 방향이 있다.
- 현재 한계: [추가 제안] 단일 주제나 감성은 약속 불이행·직원 전문성·시설·배려·대응 속도가 동시에 나타나는 VOC를 충분히 표현하지 못한다.
- 추가 기능: [추가 제안] `servqual_labels[]`, 라벨별 `confidence`, `evidence_span`, `model_version`, `reviewer_status`를 저장한다.
- 필요한 데이터: [내부 데이터 필요] 워커힐 적용 코드북, 다중 라벨 이중 검수 표본, 불일치 조정 기록이 필요하다.
- 분석 방법: [추가 제안] 신뢰성·확신성·유형성·공감성·반응성을 독립 이진 라벨로 평가하고 미해당·판단 보류를 허용한다.
- 사용자에게 제공되는 결과: [추가 제안] 모델 제안 라벨과 근거 문장, 신뢰도, 담당자 검토 완료 라벨을 구분해 표시한다.
- Human-in-the-loop: [현업 검증 필요] 라벨 충돌·저신뢰·민감 이슈는 담당자가 수정하고 수정 이력을 학습 후보와 분리 보관한다.
- MVP 구현 가능 여부: [MVP 가정] 프롬프트/규칙 기반 후보로 시작하고 Micro F1과 담당자 수정률로 검증할 수 있다.
- 기대 효과: [추가 제안] 한 VOC에 포함된 복합 품질 요소를 개선 과제 누락 없이 검토할 수 있다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] SERVQUAL 후보를 객관적 진실이나 직원 평가 점수로 표현하지 않는다.

### 개선사항 3. 유사 VOC 검색에서 반복 문제 클러스터 관리로 확장

- 논문 근거: [논문 근거] A호텔 연구는 대표 단어 코딩과 유사 유형 범주화, 반복적 비교로 빈도와 패턴을 파악했다(논문 1, PDF pp.53~55, 61~63).
- 현재 서비스 구현 내용: [현재 구현] 임베딩 기반 유사 VOC 검색과 반복 불만 탐지가 포함되어 있다.
- 현재 한계: [추가 제안] 검색 결과 목록만으로는 반복 문제의 시작·확대·조치·재발을 하나의 관리 단위로 추적하기 어렵다.
- 추가 기능: [추가 제안] `issue_cluster`에 대표 VOC, 핵심 표현, 서비스 영역, 최초·최근 발생일, 규모, 증감률, 관련 조치, 재발 상태와 병합·분리 이력을 둔다.
- 필요한 데이터: [내부 데이터 필요] 마스킹 원문, 발생일·사업장·서비스 접점, 처리·조치 ID와 노출량 분모가 필요하다.
- 분석 방법: [추가 제안] 임베딩 유사도와 메타데이터 제약을 결합하고, 담당자가 클러스터 병합·분리·대표명을 확정한다.
- 사용자에게 제공되는 결과: [추가 제안] 반복 이슈 카드, 대표 VOC, 규모 추이, 관련 사업장, 조치 후 재발 후보를 제공한다.
- Human-in-the-loop: [현업 검증 필요] 의미가 다른 이슈의 과잉 병합과 표현만 다른 동일 이슈의 분리를 담당자가 교정한다.
- MVP 구현 가능 여부: [MVP 가정] 제한된 서비스 영역에서 유사도 기준과 수동 병합 UI로 구현 가능하다.
- 기대 효과: [추가 제안] 개별 VOC 대응을 반복 문제 단위의 개선 관리로 전환한다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] 클러스터 유사성이 동일 원인이나 동일 책임자를 의미한다고 단정하지 않는다.

### 개선사항 4. 원인 후보 생성에서 근거 기반 확인 과제 제시로 확장

- 논문 근거: [논문 근거] A호텔 연구는 불만 맥락과 구체적 상황·요구를 비교해 원인 유형을 해석했지만, 단일 호텔 사례라는 한계를 가진다(논문 1, PDF pp.61~63, 85, 95~96).
- 현재 서비스 구현 내용: [현재 구현] VOC와 운영 KPI의 교차 분석으로 원인 후보를 생성한다.
- 현재 한계: [추가 제안] 후보 원인만 표시하면 상관관계가 인과로 읽히고 차별·직원 실수가 사실로 오인될 수 있다.
- 추가 기능: [추가 제안] 원인 후보마다 `voc_evidence`, `operation_evidence`, `counter_evidence`, `missing_evidence`, `fact_check_task`, `confidence`, `prohibited_conclusion`을 둔다.
- 필요한 데이터: [내부 데이터 필요] 사건 시점의 예약·안내·대기·인력·장애·점검·처리 기록과 적용 규정 버전이 필요하다.
- 분석 방법: [추가 제안] 시계열 동시 변화, 유사 과거 사례, 규정 대조를 별도 근거로 표시하고 확인 과제를 생성한다.
- 사용자에게 제공되는 결과: [추가 제안] `대기시간 증가 동반`, `인력 기록 확인 필요`, `고객 주장만 존재`, `반대 사례 존재`처럼 판단 상태가 분리된 카드가 제공된다.
- Human-in-the-loop: [현업 검증 필요] 차별·과실·안전·보상·법무·인사 관련 후보는 해당 역할의 검토 없이는 승인할 수 없다.
- MVP 구현 가능 여부: [MVP 가정] 증거 ID가 있는 근거 카드와 확인 과제는 구현 가능하며 인과 추론은 범위에서 제외한다.
- 기대 효과: [추가 제안] 원인 생성 기능을 감사 가능한 사실 확인 지원 기능으로 제한하고 신뢰도를 높인다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] `원인을 정확히 찾아낸다` 또는 `직원이 차별했다`고 표현하지 않는다.

### 개선사항 5. 권고안 생성에서 VOC 개선 백로그 관리로 확장

- 논문 근거: [논문 근거] QFD는 고객 요구를 운영·기술 요구로 치환하는 방법이며 HOQ는 그 관계를 구조화한다(논문 2, PDF pp.7~8).
- 현재 서비스 구현 내용: [현재 구현] 대응 우선순위와 권고안, 보고서 초안을 생성한다.
- 현재 한계: [추가 제안] 생성된 권고가 담당 역할·상태·기한·조치 결과·재발 검토와 연결되지 않는다.
- 추가 기능: [추가 제안] VOC 클러스터에서 개선 항목을 생성하고 담당 역할 배정, 승인·반려, 상태 변경, 조치 증적, 재발 검토를 관리한다.
- 필요한 데이터: [내부 데이터 필요] 조직 역할, 개선 유형, SLA, 업무 상태, 승인 규칙, 조치 일자와 결과 기록이 필요하다.
- 분석 방법: [추가 제안] 고객 요구→문제 현상→확인 과제→개선 과제→담당 역할→확인 KPI를 단계별로 변환한다.
- 사용자에게 제공되는 결과: [추가 제안] 개선 백로그, 책임 역할, 다음 확인일, 진행 상태, 관련 근거와 후속 재발 상태를 제공한다.
- Human-in-the-loop: [현업 검증 필요] AI는 백로그 초안만 만들고 담당자가 범위·담당·기한·완료 기준을 승인하거나 수정한다.
- MVP 구현 가능 여부: [MVP 가정] 단일 조직 체계와 제한된 상태값으로 구현 가능하다.
- 기대 효과: [추가 제안] 보고서 생성에서 끝나는 분석을 실행과 회고가 가능한 업무 흐름으로 연결한다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] 권고가 최적이거나 자동 실행될 수 있다고 주장하지 않는다.

### 개선사항 6. 이상 징후 탐지에서 설명 가능한 개선 우선순위 산정으로 확장

- 논문 근거: [논문 근거] HOQ는 고객 요구 중요도와 개선 수단의 관계 강도를 사용해 상대적 우선순위를 제시한다(논문 2, PDF p.13).
- 현재 서비스 구현 내용: [현재 구현] 이동평균·Z-score 이상 탐지와 대응 우선순위 제시가 포함되어 있다.
- 현재 한계: [추가 제안] 우선순위 요인·산식·가중치 출처가 없으면 모델의 임의 순위로 보일 수 있다.
- 추가 기능: [추가 제안] 심각도, 빈도, 반복, 증가 추세, 안전, SLA, 사업장 확산, KPI 이상 동반, 조치 후 재발, 모델 신뢰도, 담당자 조정을 각각 근거 카드로 표시한다.
- 필요한 데이터: [내부 데이터 필요] 이슈별 건수와 노출량, 안전 분류, SLA, 사업장, KPI, 과거 조치·재발, 담당자 조정 이력이 필요하다.
- 분석 방법: [추가 제안] MVP에서는 확정 가중치 없이 요인별 원점수·정규화 값·판단 보류를 보여주고, 현업이 승인한 버전형 가중치가 생긴 뒤에만 종합 점수를 계산한다.
- 사용자에게 제공되는 결과: [추가 제안] 순위와 함께 상승 요인, 데이터 결측, 가중치 버전, 담당자 조정 이유를 제공한다.
- Human-in-the-loop: [현업 검증 필요] 안전 이슈는 별도 긴급 큐로 보내고 최종 우선순위는 담당자가 승인·수정한다.
- MVP 구현 가능 여부: [MVP 가정] 요인별 점수 카드와 수동 정렬은 가능하며 임시 수치 가중치는 문서에 만들지 않는다.
- 기대 효과: [추가 제안] 왜 우선인지 설명하고 데이터 부족 시 순위 확정을 보류할 수 있다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] 빈도가 높다는 이유만으로 당연적 품질 또는 최우선 개선이라고 단정하지 않는다.

### 개선사항 7. 주간 보고서에서 개선 조치 및 재발 추적 보고서로 확장

- 논문 근거: [논문 근거] 두 논문은 개선 방안 도출의 중요성을 보여주지만 운영 시스템의 장기 재발 추적을 검증한 연구는 아니다.
- 현재 서비스 구현 내용: [현재 구현] 주간 고객경험 보고서와 대시보드가 포함되어 있다.
- 현재 한계: [추가 제안] 발생 추이와 원인 후보 보고만으로 조치가 실행됐는지, 동일 이슈가 다시 발생했는지 확인하기 어렵다.
- 추가 기능: [추가 제안] 조치 전후 VOC 발생률, 동일 유형 재발률, 평균 처리시간, SLA 준수율, 클러스터 규모, 담당자 수정률, 미해결 비율, 완료율을 보고한다.
- 필요한 데이터: [내부 데이터 필요] 조치 ID·일자·범위, 노출량 분모, 처리 상태·시간, 후속 VOC, 관찰 기간과 데이터 품질 플래그가 필요하다.
- 분석 방법: [추가 제안] 조치 전후 동일 길이의 관찰 창과 비교군 가능 여부를 표시하고, 계절·점유율·행사 등 교란 요인을 함께 제시한다.
- 사용자에게 제공되는 결과: [추가 제안] 신규·진행·완료·재발 후보·판단 보류로 구분한 주간 개선 효과 보고서를 제공한다.
- Human-in-the-loop: [현업 검증 필요] 담당자가 동일 이슈 여부와 조치 범위를 승인하고 결과 해석 문구를 검토한다.
- MVP 구현 가능 여부: [MVP 가정] 재발 후보와 기술통계 관찰은 가능하고 장기 인과효과 평가는 확장으로 둔다.
- 기대 효과: [추가 제안] 개선 활동의 폐쇄루프를 만들고 미완료·재발 이슈를 다시 백로그로 올린다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] `개선 조치로 VOC가 감소했다` 대신 `개선 조치 이후 동일 유형 VOC 감소가 관찰되었다`고 표현한다.

### 개선사항 8. Kano-QFD 기반 전략 기능은 설문 검증 후 확장

- 논문 근거: [논문 근거] Kano 유형은 긍정·부정 쌍대 설문, CSC·CDC와 정식 HOQ는 설문 응답과 관계 강도 데이터에 의존한다(논문 2, PDF pp.7, 9, 13).
- 현재 서비스 구현 내용: [현재 구현] VOC 텍스트 분석과 권고 엔진은 있으나 Kano 설문·CSC·CDC·정식 HOQ는 없다.
- 현재 한계: [현업 검증 필요] VOC 불만·칭찬 빈도만으로 Kano 유형을 정하면 방법론을 위반한다.
- 추가 기능: [추가 제안] 속성 정의, 긍정·부정 설문, 응답 품질 검사, Kano 평가표, CSC·CDC, 사업장·방문 경험별 비교, 전문가 승인 HOQ를 별도 모듈로 둔다.
- 필요한 데이터: [내부 데이터 필요] 사업장별 서비스 속성, 쌍대 설문 응답, 방문 이력의 적법한 연결, 고객 중요도, 운영 과제 관계 강도와 승인자가 필요하다.
- 분석 방법: [추가 제안] 설문으로 검증된 Kano 유형을 VOC 심각도·재발성 및 승인된 HOQ 관계와 결합한다.
- 사용자에게 제공되는 결과: [추가 제안] 설문 근거 품질 유형, CSC·CDC, 고객 요구-운영 과제 매트릭스, 승인된 전략 우선순위를 제공한다.
- Human-in-the-loop: [현업 검증 필요] 설문 문항·속성·관계 강도·우선순위는 CX·운영·조사 전문가가 검토한다.
- MVP 구현 가능 여부: [추가 제안] MVP에서는 제외하고 `설문 데이터 확보 시 예시 화면`이라는 명시가 있을 때만 빈 화면 구조를 보여준다.
- 기대 효과: [추가 제안] 단기 VOC 대응과 별도로 검증된 고객 요구 기반의 중장기 서비스 설계를 지원할 수 있다.
- 과도하게 주장하면 안 되는 내용: [현업 검증 필요] `AI가 VOC로 Kano 유형을 자동 분류한다`, `불만이 많아 당연적 품질이다`, `칭찬이 많아 매력적 품질이다`라는 표현을 금지한다.

## 7. VOC-to-Action 처리 흐름

```text
VOC 수집
→ 개인정보 비식별화 및 분석 차단 검사
→ 서비스 영역·SERVQUAL 다중 라벨 후보
→ 기대 상태·실제 경험·갭·요구 조치 추출
→ 유사·반복 VOC 클러스터링
→ 운영 KPI·처리 이력·규정 버전 연결
→ 심각도·재발성·SLA·증가 추세 평가
→ 고객 요구사항과 확인 필요 사실 구조화
→ 운영 개선 과제 후보와 관계 근거 제시
→ 담당자 승인·수정
→ 개선 백로그 등록·상태 관리
→ 조치 결과·증적 기록
→ 동일 VOC 재발 후보 확인
→ 주간 개선 효과 보고
```

[추가 제안] 서비스의 핵심 전환은 `원인·권고 문장 생성`에서 `근거가 있는 확인 과제와 승인 가능한 개선 백로그`로 바꾸는 것이다.

### 간소화 VOC-to-Action 관계표

| 고객 요구사항 | 운영 개선 과제 후보 | 관계 강도 후보 | 근거 | 담당자 검토 |
|---|---|---:|---|---|
| [MVP 가정] 체크인 대기 감소 | [추가 제안] 피크 시간대 인력·동선·사전등록 절차 확인 | 5 | [MVP 가정] 대기시간 KPI와 동일 시간대 VOC 증가가 함께 관찰된 데모 시나리오 | 필수 |

[MVP 가정] 위 표의 5·3은 논문에서 사용한 HOQ 관계 척도를 설명하기 위한 합성 예시이며 실제 워커힐 관계 강도나 우선순위가 아니다.

[현업 검증 필요] 운영에서는 AI가 관계 강도 후보와 근거만 제시하고 담당자가 승인한 값만 집계하며, 정식 HOQ라는 명칭은 설문 중요도와 관계 매트릭스가 갖춰진 뒤 사용한다.

## 12. 적용하면 안 되는 논문 결과

1. [논문 근거] 마을호텔 표본에서 객실 보안이 당연적 품질로 분류된 결과는 워커힐의 Kano 유형이 아니다.
2. [논문 근거] 마을호텔 표본에서 객실 물온도가 일원적 품질로 분류된 결과는 워커힐의 Kano 유형이 아니다.
3. [논문 근거] 객실 어메니티·향기·온도 등 중요도 순위는 워커힐 우선순위로 사용할 수 없다.
4. [논문 근거] 노스텔지어 프로그램·사랑방 문화·마을호텔 관광시설·마을호텔 VOC 관리 순위는 워커힐 개선 과제가 아니다.
5. [논문 근거] A호텔의 칭찬·불만 빈도, 발생 장소, SERVQUAL 비중과 시설 노후화 등 세부 결과는 워커힐 현황이 아니다.
6. [논문 근거] A호텔에서 도출한 누적형·차별대우·직원 실수·주관/객관 기준은 워커힐에서 검증된 원인 분포가 아니다.
7. [현업 검증 필요] 고객 불만이 많다는 사실만으로 당연적 품질이라 분류하거나 칭찬이 많다는 사실만으로 매력적 품질이라 분류하지 않는다.
8. [현업 검증 필요] VOC와 운영 KPI가 같은 시점에 변했다는 이유로 운영 KPI가 VOC의 원인이라고 단정하지 않는다.
9. [현업 검증 필요] AI가 직원 과실·차별·고객 진술의 진실성·법적 책임·보상 의무를 확정하지 않는다.
10. [현업 검증 필요] 고객 만족도·재방문율·매출 상승·인건비 절감·처리시간 감소·최적 개선안 제공은 직접 검증 없이는 문서에 넣지 않는다.

## 13. 서비스 핵심 차별화

1. [추가 제안] **MVP 차별화:** VOC에서 고객이 기대한 상태와 실제 경험의 차이, 요구 조치와 확인할 사실을 구조화하고, 운영 지표·처리 이력의 근거를 붙여 담당자가 승인할 개선 과제로 전환하는 플랫폼이다.
2. [추가 제안] **실행 차별화:** 유사 VOC를 검색하는 데서 멈추지 않고 반복 이슈 클러스터를 개선 백로그와 연결하여 담당 역할·조치 상태·동일 이슈 재발 후보를 주간 단위로 추적한다.
3. [추가 제안] **확장 차별화:** 별도 쌍대 설문으로 검증한 Kano 유형과 CSC·CDC, 전문가가 승인한 QFD·HOQ 관계를 VOC 심각도·재발성 분석과 결합해 중장기 서비스 설계를 지원한다.

[현재 구현] 첫째와 둘째 문장 중 분류·검색·운영 교차·원인 후보·보고는 현재 기획에 포함되어 있으나, 갭 구조화·백로그·재발 상태·관계표는 추가 구현이 필요하다.

[추가 제안] 셋째 문장은 현재 기능이 아니라 데이터와 설문이 확보된 이후의 확장 방향이다.

## 14. 결론

[추가 제안] 현재 서비스에 이미 반영된 논문 요소는 VOC 코딩, 서비스 영역·SERVQUAL 후보 분류 방향, 유사·반복 분석, 감정·긴급도, 운영 KPI 교차, 원인 후보와 권고안이다.

[추가 제안] 현재 구현안은 문제 탐지까지는 비교적 구체적이지만, 고객 요구 정의→운영 과제 변환→담당자 승인→백로그→조치 결과→재발 확인의 실행 폐쇄루프가 부족하다.

[추가 제안] 핵심 추가 기능은 요구사항·서비스 갭 추출기, SERVQUAL 다중 라벨, 반복 이슈 클러스터 관리, 근거·반대 근거·확인 과제, 설명 가능한 우선순위 카드, 개선 백로그, 조치·재발 보고다.

[MVP 가정] MVP는 위 기능을 제한된 합성/승인 데이터와 담당자 승인 흐름으로 구현할 수 있으며, 정식 Kano·CSC·CDC·QFD·HOQ와 장기 인과효과 평가는 확장으로 분리한다.

[내부 데이터 필요] 실운영 적용을 위해 가장 먼저 확인할 항목은 VOC 원문·발생 시각·사업장·답변·처리·SLA·조치 ID, 운영 KPI의 시간·접점 키, 노출량 분모, 정책 버전, 담당 역할·승인 절차다.

[현업 검증 필요] AI 자동 판단 금지 영역은 차별·과실·법적 책임·보상, 고객 진술의 진실성, 직원 평가, 인과효과, Kano 유형, 최종 우선순위와 고객 답변이다.

[현업 검증 필요] 이 문서의 최종 제안은 연구 방법의 전이이며 A호텔·마을호텔 연구 결과의 전이가 아니다.

### 15.2 Gate 최종 판정

[논문 근거] 방법론으로 이전한 것은 VOC 필드, 다중 코딩, 반복 범주화, 기대-지각 갭, 고객 요구→운영 과제 변환, 관계 매트릭스의 검토 절차다.

[현업 검증 필요] 결과 전이를 금지한 것은 A호텔 빈도·원인 분포와 마을호텔 Kano 유형·속성 순위·개선 순위다.

[MVP 가정] MVP에서 가능한 것은 후보 추출·근거 제시·클러스터·우선순위 요인 카드·백로그·승인·재발 관찰이다.

[내부 데이터 필요] 설문과 승인 관계 데이터가 필요한 것은 Kano 유형·CSC·CDC·정식 QFD·HOQ다.

### 15.3 기존 문서 변경 기준

[현재 구현] 기존 문서의 수익관리·다이내믹 프라이싱·식자재 수요예측은 이미 `프로젝트가 해결하지 않는 영역`과 `MVP 제외`로 이동되어 있어 삭제 상태를 유지한다.
