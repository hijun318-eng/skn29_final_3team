# SensePlace — 호텔 VOC·운영 지원 플랫폼

**SK네트웍스 Family AI 캠프 29기 3팀** · 공식 서비스명 **SensePlace** · 호텔 VOC·운영 이슈 분석 에이전트

> 단일 호텔의 합성 조식 운영 데이터와 합성 VOC를 이용해, 권한 있는 사용자의 제한된 자연어 조회(기능 A)와 이상 감지→근거 조사→보고서 검토(기능 B)를 제공하는 내부 운영 지원 PoC. AI는 원인을 확정하지 않고 자동 실행하지 않으며 시스템 종료 지점은 호텔 관리자의 승인·보류·반려다.
> 작성 기준일 2026-07-21 · v2.6 · 데이터: 합성 운영 데이터+합성 VOC 전용 · 추적 요구사항 **62개**

> 제품·Baseline 기준은 `docs/markdown/SensePlace_기획서_초안.md`를 따른다. 기존 62개 ID는 추적성을 위해 보존하되, 아래 현재 범위와 각 상세 항목의 범위 표시를 동일하게 적용한다. 화면설계서는 이번 동기화 범위에서 제외한다.

세 범주의 합계는 기존 요구사항 62개와 일치한다. 62개는 추적용 전체 목록이며 모두가 확정 구현 범위라는 뜻은 아니다. 실제 내부 데이터·실시간 연동·자동 조치·다호텔 운영은 프로젝트 비목표다.

Baseline 43개는 우선순위상 Must 39개와 Should 4개(`UI-004`,`NFR-001`,`NFR-005`,`OPS-001`)로 구성된다. 8월 13~14일 Gate에서는 기능 A·B의 실행 경로와 최소 진실성·보안·회귀 기준을 차단 조건으로 검증한다. `UI-004`의 단위·기간 표시는 Gate 차단 기준이고 색상 외 접근성 강화만 비차단이다. `NFR-001` 성능 목표와 `OPS-001` 운영 이력 조회의 최종 검증도 Gate 비차단으로 7주차까지 완료한다. `TST-003`은 5주차 Gate 자동 채점, 6~7주차 회귀, 8주차 Golden Path 연속 5회로 단계화한다.

## 📌 핵심 원칙

1. 핵심은 **VOC×운영지표 교차분석**(단순 감성분석 아님).
2. **수치 계산과 이상 판정은 SQL·Python·규칙, LLM은 질문 해석 또는 근거 기반 서술에 실행당 최대 1회**만 담당한다.
3. **PostgreSQL `metric_catalog` 기반 경량 시맨틱 카탈로그로 지표·동의어·허용 차원·조회 범위·권한을 고정**한다. 정식 온톨로지·GraphDB는 도입하지 않으며 VectorDB는 독립 실험 전용이다.
4. 모든 결과에 **근거 수치·기간·리뷰** 첨부, 원인은 **후보**로만 표현.
5. 실제 개인정보는 생성·수집·저장하지 않는다. 합성 데이터에 개인정보성 값이 포함되지 않았는지 검증하며 최종 결정·조치는 운영자가 수행한다.

요구사항의 **담당 트랙·리드**는 실행 주도 역할이며, Baseline 항목의 최종 검토·결과 책임은 `Baseline 43건 공통 수용 추적`의 Owner를 따른다.

## 🎯 목표와 비목표

**목표:** 전량 합성 VOC·운영 데이터로, 권한 기반 대화형 분석과 결정론적 이상 감지→근거 조사→주간 보고서→관리자 결정을 연결하는 SensePlace Baseline을 구현한다.

**비목표:** 인과 확정 · 조치 자동 실행 · 실시간 고객 응대 · 완전 자유형 BI · 실제 PMS·POS·CRM 연동 · 상용 다중 호텔 운영.

```
전량 합성 데이터 생성→적재 전 validation→PostgreSQL 적재→적재 후 analytical Gate

  → 반례 세트 v2 21건·intent matrix Gate→Golden Path 연속 5회
```
