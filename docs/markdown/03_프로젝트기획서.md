## 1. 프로젝트 주제

### 1.1 한 줄 정의

SensePlace는 단일 호텔의 합성 조식 운영 데이터와 합성 VOC를 이용해, 권한 있는 사용자의 제한된 자연어 조회를 지원하고 조식 대기 이상을 자동 감지하여 **관측 사실·원인 후보·반대 근거·현장 점검 항목·대응 옵션·주간 보고서 초안**을 제공하는 내부 운영 지원 PoC다.

### 1.2 프로젝트 전제와 표현 경계

- 기업 내부 PMS·POS·CRM·근태·VOC 시스템과 실제 내부 데이터에는 접근하지 않는다.
- 실제 호텔의 운영 문제, 인력 부족, VOC 증가, 도입 효과를 주장하지 않는다.
- 합성 데이터에서 함께 변한 지표는 원인이 아니라 확인이 필요한 후보로만 표현한다.
- 모든 화면·API·보고서에 `SYNTHETIC DEMO DATA`와 데이터·스키마·생성기 버전을 표시한다.
- 실제 호텔 조직명과 권한을 안다고 가정하지 않으며 역할은 데모 정책으로만 정의한다.

### 1.3 사용자와 시스템 경계

| 구분 | 역할 | 시스템 안의 완료 결과 |
|---|---|---|
| 호텔 관리자 | 전체 합성 운영 현황·이슈·보고서 검토 | 보고서 승인·보류·반려 결정 |
| F&B 관리자 | 조식 운영·인력·VOC 범위 질의·이슈 확인 | 허용 범위의 분석 결과와 근거 확인 |
| 객실 관리자 | 객실 집계·객실 VOC·제한된 조식 수요 확인 | 권한 범위 결과 확인 또는 거부 확인 |
| 시스템 밖 | 본사 경영진·실제 운영 담당자 | 본사 전송·실제 조치·성과 판단은 수동 업무 |

최종 시스템 종료 지점은 호텔 관리자의 승인·보류·반려다. 고객 메시지, 보상, 인력 배치, 티켓 생성과 본사 전송은 자동 실행하지 않는다.

### 1.4 목표

1. 고정 seed와 버전이 있는 합성 데이터로 정상·이상·충돌·결측 상황을 재현한다.
2. 역할별 허용 범위를 서버에서 강제하는 자연어 분석을 제공한다.
3. 적재 전 validation과 적재 후 analytical Gate를 통과한 데이터에서 결정론적 규칙으로 이상징후를 감지한다.
4. 운영지표와 VOC의 관측 사실·원인 후보·반대 근거·부족한 데이터를 구분한다.
5. 수치는 SQL·Python·규칙으로 계산하고 LLM은 질문 해석 또는 근거 기반 서술에 실행당 최대 1회 사용한다.
6. 주간 보고서 초안과 버전·승인 이력을 연결하고 실패 시 성공한 수치 결과를 보존한다.

### 1.5 비목표

- 실제 PMS·POS·CRM·근태·VOC 연동과 실제 온라인 리뷰 수집
- 실제 고객·직원 개인 데이터, 실시간 스트리밍과 다호텔 비교
- 완전 자유형 BI와 운영 조치 자동 실행
- 본사 계정·자동 전송과 고객 대상 챗봇
- GraphDB·GraphRAG·정식 OWL ontology·장기 memory·Agent swarm의 Baseline 의존성화
- 수요·매출·이탈 예측과 실제 비용 절감 효과 확정

## 2. 문제 정의

공개자료만으로는 그랜드 워커힐 서울의 실제 VOC 분산, 수작업 보고, 처리 지연이나 분석 병목을 확인할 수 없다. 따라서 아래는 제품 필요성을 확정하는 사실이 아니라 합성 데이터 PoC와 후속 실데이터 파일럿에서 검증하거나 기각할 가설이다.

| ID | 문제 가설 | 프로젝트 검증 | 기각·축소 조건 |
|---|---|---|---|
| H1 | 폐쇄형 8 intent 자연어 조회를 권한 범위에서 안전하게 실행할 수 있다. | 8 intent×3발화×3역할의 기대 허용·거부와 수치 일치 | 한 건이라도 권한 우회·공격 SQL 실행 또는 지원 질의 수치 불일치 |
| H2 | 이상 감지 후 운영지표와 VOC 근거를 재현 가능하게 연결할 수 있다. | 반례 세트 v2·복수 seed의 탐지·근거 연결 | 차단 결함 잔존, 정상 scenario 경보, 결측인데 원인 후보 생성 |
| H3 | 관측 사실·원인 후보·반대 근거를 결과 구조에서 분리할 수 있다. | 금지 인과 표현 0건, evidence 없는 주장 0건 | 원인 단정·반대 근거 누락·허위 근거 발생 |
| H4 | LLM 장애에도 핵심 수치와 검토 가능한 주간 DRAFT를 유지할 수 있다. | 수치 일치, `PARTIAL` template, decision 이력 | LLM timeout으로 수치 소실 또는 자동 승인·자동 조치 발생 |

### 2.1 주요 위험과 대응

| 위험 | 대응·Gate |
|---|---|
| 합성 결과를 실제 호텔 현황처럼 표현 | 모든 결과에 합성·가상 기준일 표시, 실제 문제·성과 주장 금지 |
| 생성 수식과 탐지 규칙의 자기검증 | 설정과 규칙을 분리하고 교차 리뷰, 정답 manifest를 런타임에서 차단 |
| VOC·운영 공통 키 또는 시간 정합성 부족 | 서비스 구역·기간·timezone 정합성 검증 실패 시 `NEEDS_DATA` |
| LLM 환각과 수치 오류 | 수치 결정론 계산, 근거 ID 없는 문장 차단, 실패 시 `PARTIAL` |
| 권한 우회와 SQL 위험 | Django scope 원본, allowlist, parameter binding, read-only role, timeout·row limit |
| Django·FastAPI 이중 구조의 일정 위험 | 08/14 Gate에서 통합 복잡도 측정, 미통과 시 확장 포기와 범위 재결정 |
| 문서 간 범위 재확대 | 요구사항·WBS에서 Baseline·독립 실험·승인 후 확장·범위 밖을 구분하고 Gate 통과 전 확장 금지 |

### 3.1 기대 가치와 측정

| 가치 경로 | 제공 가치 | PoC 측정 항목 |

매출 증가, 비용 절감과 실제 업무시간 단축은 본 PoC에서 산정하지 않는다. 실제 기준선과 비식별 데이터가 확보된 파일럿에서만 측정한다.

| 최종 발표에서 가능한 주장 | 금지하거나 조건을 붙일 주장 |
|---|---|
| 합성 scenario에서 제한된 질의·권한·SQL 공격 차단을 재현했다 | 자유형 자연어 BI를 완성했다 |
| 합성 조식 대기 이상에서 관측·후보·반대 근거·점검 항목을 연결했다 | 실제 호텔의 원인을 발견하거나 운영 문제를 입증했다 |
| LLM 실패에도 결정론적 수치와 template DRAFT를 유지했다 | 실제 업무시간·비용을 절감했다 |
| 단일 호텔 PoC의 기술 실현 가능성과 실패 경계를 확인했다 | 다호텔·대용량·실시간·상용 운영에 그대로 적용 가능하다 |

### 3.2 레퍼런스와 차별화

| 참고 사례 | 참고 영역 | 이번 프로젝트의 초점 |
|---|---|---|
| Shiji ReviewPro | 호텔 리뷰·설문·운영 workflow 참고 | 합성 VOC와 운영지표를 동일 조건으로 연결 |
| TrustYou | 리뷰 통합·감성·주제 분석 참고 | 한국어 VOC 구조화와 근거 문장 |
| Medallia | 역할별 CX 리포트와 closed-loop 참고 | 자동 실행 없는 검토·승인 경계 |
| Qualtrics | 경험·운영 데이터 연결 참고 | 권한 기반 자연어 조회와 결정론적 분석 |

위 항목은 시장 규모나 실제 도입 효과의 근거가 아니라 공개 제품 범주를 파악하기 위한 참고 사례다. 최종 발표에서 구체 기능을 비교 인용할 경우 해당 기업의 공식 자료와 확인일을 별도로 기록한다.

기존 서비스를 복제하지 않고 **합성 데이터 기반 호텔 운영 분석, 근거·반대 근거, 비확정 원인 후보, Human-in-the-loop**를 검증한다.

### 3.3 제공 형태와 실제 도입 제안

프로젝트 기간에는 단일 호텔용 내부 웹 PoC로 제공한다. 실제 도입은 다음 단계형 파일럿으로만 제안한다.

1. 비식별 표본 데이터로 schema·연결 키·집계 주기를 검증한다.
2. 실제 분포에 맞춰 임계값을 재보정하고 정밀도·오탐률을 측정한다.
3. 수작업 보고와 시스템 보조 보고의 시간·누락·수정률을 비교한다.

각 단계가 실패하면 효과를 주장하지 않고 범위를 축소하거나 중단한다. 상용화 방식과 가격은 실제 데이터 적합성·업무 효과·보안 비용을 확인한 뒤 검토한다.

## 4. 시스템 구성 기획

### 4.1 서비스 흐름

```mermaid
flowchart TB
    U[호텔 내부 사용자] --> D[Django 로그인·역할 확인]
    D --> H[SensePlace 홈]
    H --> QA[기능 A: 자연어 질문]
    QA --> QR[표·차트·설명·근거]

    G[합성 데이터 생성] --> PV[적재 전 validation]
    PV --> B[PostgreSQL 적재 batch]
    B --> QG[적재 후 analytical Gate]
    QG --> DT[결정론적 이상 감지]
    DT --> IA[기능 B: Incident 분석]
    IA --> RP[이슈 브리프·주간 보고서 DRAFT]
    RP --> RV[호텔 관리자 검토]
    RV --> DC[승인·보류·반려]
```

두 경로는 하나의 홈과 근거·보고서 이력에서 연결되지만 강제로 합치지 않는다. 기능 A는 질문 결과로 종료하고 기능 B는 호텔 관리자 결정으로 종료한다.

### 4.2 논리 아키텍처와 책임

| 계층 | 책임 | 금지 |
|---|---|---|
| Frontend | 역할 선택, 대화·차트·이슈·보고 검토 | 권한 최종 판단, 수치 계산 |
| Django + DRF | 외부 API, 인증·RBAC, scope 원본, job·report·decision·감사 이력 | 분석 view 직접 계산, LLM 판정 |
| Django worker | 비동기 실행·상태·retry·idempotency 관리 | 브라우저 직접 노출 |
| 합성 데이터 pipeline | 적재 전 schema·PII·행 정합 validation | 실패 batch 적재, 원문 임의 치환 |
| FastAPI | 적재 후 analytical Gate·규칙 감지, query pipeline, Incident workflow, SQL Guard, 근거·설명 구성 | 로그인 원본, 승인 table 쓰기 |
| PostgreSQL | 합성 analytics view, run·evidence·report 저장 | LLM의 임의 직접 접근 |
| External LLM API | 질문 해석 또는 근거 기반 설명·보고 문장, 실행당 최대 1회 | 이상 판정, KPI 계산, 외부 시스템 쓰기 |

호출 방향은 `Client → Django → Worker → FastAPI → PostgreSQL/LLM`으로 고정한다. Browser는 FastAPI를 직접 호출하지 않고 FastAPI는 Django가 검증한 `scope_snapshot`만 신뢰한다.

#### 시연·배포 환경 원칙

| 환경 | 현재 기준 | 미결정·fallback |
|---|---|---|
| 중간발표 | 로컬 브라우저의 비연동 fixture 목업 | 합성·목업 표시, 녹화본 예비 준비 |
| Baseline 검증 | 고정 합성 dataset과 동일한 버전의 로컬 통합 환경 | secret은 환경변수, 외부 LLM 실패 시 template 결과 |
| 최종발표 | 6주차 시스템 구성도에서 단일 시연 장비 또는 배포 주소를 확정 | 네트워크·LLM 장애 시 결정론 수치 화면과 시연영상 사용 |

외부 공개 배포와 장기 운영은 Baseline 완료 조건이 아니다. 최종 시연 환경은 6주차에 확정하고 7주차 서비스 테스트와 8주차 Golden Path에 동일하게 사용한다.

### 4.3 Job·보고서 상태

```text
PENDING → RUNNING → SUCCEEDED
                  ↘ PARTIAL
                  ↘ NEEDS_DATA
                  ↘ FAILED
```

```text
DRAFT → APPROVED
      ↘ ON_HOLD → 새 DRAFT version
      ↘ REJECTED
```

사용자 요청은 비동기 job으로 처리하고 UI가 상태를 조회한다. timeout·retry 수치는 시연 환경 측정 후 구현 설계에서 정하되 중복 분석과 승인된 보고서 덮어쓰기는 금지한다.

### 4.4 중간발표 화면 기준

| 순서 | 화면 | 핵심 내용 |
|---:|---|---|
| 1 | 가상 로그인·역할 선택 | 호텔·F&B·객실 관리자, 합성·목업 고지 |
| 2 | 운영 홈 | 질의 진입, 합성 데이터 상태, 이상징후 카드 |
| 3 | 대화형 분석 | 질문, 목업 처리 단계, 표·차트·설명, 권한 범위 |
| 4 | 이상징후 상세 | 감지 지표·비교 기간·데이터 상태 |
| 5 | 이슈 브리프 | 관측 사실·원인 후보·반대 근거·추가 확인 |
| 6 | 주간 보고서 | DRAFT, 근거 ID, 한계, 승인·보류·반려 |

중간발표 fixture는 역할·상태·근거·합성 데이터 표시라는 최소 공통 항목을 사용한다. 상세 API 형식을 확정한 것처럼 표현하지 않고, 실제 backend·DB·LLM 연결 결과처럼 보이게 하지 않는다.

### 4.5 최소 연동 경계

| 경계 | 기능 | 기획 단계 필수 합의 |
|---|---|---|
| Browser→Django | 로그인·질문·상태 조회·이슈·보고서 검토 | 역할·scope와 사용자 작업 이력은 Django가 관리 |
| Django→FastAPI | 품질 검사·감지·질문 분석·Incident 조사 요청 | Django가 확인한 역할·데이터 버전만 전달 |
| FastAPI→DB | 합성 analytics view 조회 | read-only와 허용 지표·차원 적용 |
| Django 보고 흐름 | DRAFT 조회·승인·보류·반려 | 결정 이력과 기존 승인본을 보존 |

실행·역할·합성 데이터 버전·상태·근거를 역추적할 수 있어야 한다. 구체 endpoint·필드 타입·오류 코드는 구현 설계에서 정한다.

### 4.6 Baseline과 실험·확장 경계

| 구분 | 범위 |
|---|---|
| Baseline 실행 경로 | semantic catalog, 결정론적 SQL builder·Guard, 기능 A 명시적 pipeline, 기능 B Incident workflow, 2단계 품질 검사·규칙 감지, 승인 workflow |
| 독립 실험 | VOC 분류, VOC 군집, pgvector 유사 VOC 검색, sLLM 비교, 단일/다역할 Agent 비교, 합성 평가 구간별 모델 편차 점검 |
| 승인 후 확장 | Celery·Redis, MCP read-only adapter, RAG, 정식 ontology·GraphDB, 추가 시설·질문·이상 규칙 |
| 프로젝트 밖 | 실제 내부 시스템·실데이터·자동 조치·다호텔 상용 운영 |

실험 코드는 실행 경로가 의존하지 않도록 분리한다. 실험 실패가 Baseline 완료를 막지 않으며 실험 결과를 화면에 노출하면 참고 정보로만 표시한다.

## 5. 모델링 계획

### 5.1 실행 경로 모델링

| 영역 | 방식 | 검증 |
|---|---|---|

### 5.2 실험 트랙

| 실험 | 비교 | 실행 경로와의 관계 | 주요 지표·산출 시기 |
|---|---|---|---|
| VOC 분류 | TF-IDF+선형 모델 vs 한국어 사전학습 모델 | 런타임 판정에 미사용 | Macro F1·클래스별 F1·오류 유형, 4주차 |
| VOC 군집 | KMeans/HDBSCAN 후보 vs 규칙 주제 정답표 | 결과를 원인 판정에 미사용 | 군집 품질·안정성·해석 한계, 4주차 |
| 유사 VOC 검색 | pgvector 임베딩 검색 | 이슈 브리프 참고 섹션 후보 | Recall@K·근거 적합률, 4주차 |
| sLLM | API LLM과 동일 hold-out 입력 비교 | 런타임 기본은 API LLM | 품질·지연·비용·근거 준수, 5주차 |
| Agent 구성 | 단일 구성 vs 계획·조사·작성 역할 | Baseline 논리 경로와 분리 | 정확성·재현성·지연·실패율, 6주차 |
| 모델 편차 | 정의 가능한 합성 template family·표현 구간 | Baseline Gate 비차단 독립 점검 | 구간별 성능 편차·표본·한계, 6주차 |

### 5.3 반례 검증 범주

Baseline Gate에서는 **반례 세트 v2 21건**을 검증한다. 기존 정상·동시 이상·단독 이상·결측·권한·비가산 재집계·중복 batch·FastAPI/LLM timeout·근거 외 주장·승인 전 확정 노출·목업 오인에 `HIGH_DEMAND_NORMAL_CAPACITY`, `CAPACITY_DROP_WITHOUT_STAFF_DROP`, VOC 접수 지연, shift 경계 불일치, SQL 공격 suite를 추가한다. SQL 공격 suite는 `UNION`, 주석, 다중 statement, 금지 함수·catalog 접근, scope 우회를 포함한다. 세부 입력·기대 결과와 차단 결함 등급은 `SensePlace_기획서_초안.md` §12.4와 요구사항 `TST-001`,`TST-003`에서 관리한다.

## 6. 데이터 수집 전략

### 6.1 데이터 원칙

Baseline과 모델 실험 입력은 `synthetic_ops`와 `synthetic_voc`로 고정한다. 공개 자료는 호텔 범위와 도메인 용어를 참고하는 데만 사용하며 실제 호텔 운영 수치나 실제 VOC로 표현하지 않는다. 공개 리뷰 데이터셋과 실제 고객 데이터는 Baseline 및 모델 실험에 사용하지 않는다.

### 6.2 데이터 도메인

| 도메인 | 목적 | 제외 정보 |
|---|---|---|
| 객실 운영 | 조식 예상 수요의 상위 맥락 | 실제 요금·고객명·예약번호 |
| 조식 운영 | 도착·처리량·대기 이상 감지 | 실제 영업장명·결제 정보 |
| 조식 인력 | 운영지표와 함께 확인할 후보 근거 | 직원명·근태 사유·개인 평가 |
| VOC | 대기 이슈의 고객 관측 근거 | 실제 리뷰 원문·작성자·계정 |

Baseline 서비스 구역은 실제 시설명 대신 `GW_BREAKFAST_DEMO`를 사용한다. 매출·ADR·RevPAR·날씨·행사·온라인 crawling은 핵심 시나리오 통과 후 필요성을 확인한다.

요일·시간대·객실 판매·조식 대상 인원·도착 집중은 합성 분포에 반영하되, 성수기·단체 고객·행사·날씨·시설 고장처럼 수집하지 않은 요인은 관측 불가능한 혼동 요인으로 명시한다. 이 경우 시스템은 인력 부족 같은 단일 원인을 확정하지 않고 추가 현장 확인 또는 `NEEDS_DATA`를 제시한다.

### 6.3 생성·적재 흐름

```mermaid
flowchart LR
    A[공개 사실·프로젝트 가정 분리] --> B[고정 seed 생성기]
    B --> C[정상 12주 데이터]
    C --> D[이상·충돌·결측 scenario 주입]
    D --> E[적재 전 validation]
    E --> F[PostgreSQL 적재]
    F --> G[적재 후 analytical Gate]
    G --> H[정답 manifest 회귀 test]
```

탐지 규칙은 생성 수식을 참조하지 않고 관측 창과 비교 창의 통계만 사용한다. generator config와 rule은 분리하고 정답 manifest는 테스트 전용 경로에 두어 Agent가 접근하지 못하게 한다.

### 6.4 기획 단계 최소 데이터 구조

| 구분 | 핵심 테이블·항목 |
|---|---|
| metadata | `dataset_manifest`, `dim_date`, `dim_service_area` |
| facts | `fact_rooms_daily`, `fact_breakfast_15m`, `fact_breakfast_daily`, `fact_staff_shift`(시작·종료 시각), `fact_voc`, `fact_voc_topic` |
| platform | `metric_catalog`, `role_scope`, `query_run`, `analysis_run`, `evidence`, `report`, `report_decision` |

모든 timestamp는 UTC로 저장하고 Asia/Seoul로 표시한다. 결과에는 관측 창·비교 창·집계 단위·표본 수·timezone을 포함한다. p90 같은 비가산 지표는 15분 bucket 평균으로 재집계하지 않으며 허용 grain을 `metric_catalog`에서 제한한다.

이 표는 구현 전 합의할 데이터 개념과 연결 관계만 제시한다. 상세 컬럼·PK/FK·허용값은 DB/저장소 설계 문서에서 정하며 별도 기획 계약 산출물로 만들지 않는다. `fact_voc`에는 channel·rating·stay_date를 두지 않으며 발생 시각이 없으면 `received_at`과 낮은 시간 정렬 품질을 명시한다.

### 6.5 공통 메타정보와 2단계 품질 검사

필수 메타정보는 `is_synthetic`, `dataset_version`, `schema_version`, `generator_version`, `scenario_id`, `seed`, `virtual_as_of_date`, `data_cutoff`다.

**적재 전 validation**은 다음을 검사하고 실패 batch를 적재하지 않는다.

1. schema·형식·필수값과 PK 중복·FK 고아 0건
2. `occurred_at ≤ received_at`, shift 시작·종료와 15분 bucket 경계 정합
3. PII·secret과 실제 고객·직원 식별값 0건
4. 객실 재고·판매·투숙객·조식 대상 인원의 행 단위 범위 정합성

**적재 후 analytical Gate**는 다음을 검사하고 실패 시 분석 상태를 `NEEDS_DATA`로 둔다.

1. 15분 도착 합계와 일 합계, 대기·처리량·대기열의 비음수 조건
2. 필수 시간 bucket 완전성과 분석 기간 coverage
3. VOC·운영지표의 서비스 영역·기간·timezone·grain 정렬
4. 비가산 지표의 원자료 재계산 가능성과 집계 결과 정합

### 6.6 필수 시나리오

| scenario | 기대 결과 |
|---|---|

### 7.2 협업·승인 원칙

- 상세 담당·일정·현황은 `02_WBS.md`를 따른다.
- 데이터 항목·시간 단위·권한·상태의 최소 합의를 3주차 병행 작업의 기준으로 사용하고 변경 시 fixture·구현 영향분을 함께 검토한다.
- 생성기와 탐지 규칙은 서로 다른 담당자가 교차 검토한다.
- Baseline Gate 통과 전에는 선택 확장을 실행 경로에 추가하지 않는다.
- 발표자료·대본·시연영상·리허설·발표 책임은 정승에게 둔다. 다른 팀원은 요청받은 기술 근거와 결과만 제공한다.
- 박준희는 B·D의 핵심 구현을 대신하지 않고, 두 트랙이 겹치는 구간의 요구사항·연동 기준·scenario·정답 manifest·수용 기준·테스트 조율·Gate 결함 분류·결과 보고를 책임져 김재홍·윤대성의 공통 업무 부담을 줄인다.
- 범위 변경 승인 시 요구사항·WBS·검증 기준을 같은 변경 묶음으로 동기화한다. 화면설계서는 별도 요청 전까지 수정하지 않는다.
