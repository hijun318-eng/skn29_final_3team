# 호텔 VOC·운영 이슈 분석 AI Agent

> 글로벌 호텔 체인이 이미 활용 중인 VOC·CX·AI 플랫폼을 참고해,  
> 고객 불만과 운영 데이터를 연결하는 내부 운영자용 AI Agent를 제안합니다.

---

## 0. AI가 이 문서를 읽을 때 반드시 지켜야 할 규칙

이 문서는 프로젝트의 **기획·구현·데모 범위에 대한 기준 문서**입니다.

AI 또는 개발자는 작업 전 아래 원칙을 우선 적용해야 합니다.

1. 이 프로젝트는 단순 리뷰 요약 서비스가 아니다.
2. 핵심은 **VOC와 운영지표의 교차분석**이다.
3. 수치 계산은 SQL, Pandas 또는 명시적 규칙으로 수행한다.
4. LLM은 감성·주제 분류, 질문 해석, 설명 생성에 사용한다.
5. LLM이 원인을 확정해서는 안 된다.
6. 분석 결과에는 반드시 근거 수치와 리뷰 원문이 포함돼야 한다.
7. 실제 호텔 데이터가 아닌 합성 운영 데이터를 사용한다는 한계를 명시한다.
8. 문서에 없는 기능을 임의로 확장하지 않는다.
9. 외부 서비스의 기능이나 호텔별 사용 여부를 추정하지 않는다.
10. 레퍼런스가 오래됐을 가능성이 있으면 공식 자료를 다시 확인한다.

---

## 1. 프로젝트 정의

### 1.1 한 줄 정의

호텔 VOC와 운영 데이터를 함께 분석해,  
부정 VOC가 증가한 서비스의 **원인 후보와 점검 방향**을 제공하는 내부 운영자용 AI Agent입니다.

### 1.2 해결하려는 질문

- 최근 부정 VOC가 증가한 서비스는 무엇인가?
- 같은 기간 어떤 운영지표가 함께 변했는가?
- 운영자가 우선 점검해야 할 원인 후보는 무엇인가?
- 어떤 대응 방안을 검토할 수 있는가?

### 1.3 핵심 차별점

기존 서비스:

> 고객이 무엇을 불편해했는지 보여줌

우리 서비스:

> 고객 불만이 증가한 기간에 함께 변한 운영지표를 찾아 원인 후보와 점검 방향을 설명함

### 1.4 비목표

이 프로젝트는 아래 기능을 목표로 하지 않습니다.

- 실제 호텔 문제의 인과관계 확정
- 법적·경영적 판단 자동화
- 실시간 고객 응대
- 완전 자유형 자연어 분석
- 대규모 글로벌 호텔 운영 플랫폼 복제
- 실제 PMS·CRM·OTA 전면 연동

## 4. 글로벌 호텔 체인의 VOC 활용 흐름

```text
리뷰·설문·메시지·통화 수집
        ↓
감성·주제·불만 유형 분석
        ↓
호텔·시설·기간별 이슈 탐지
        ↓
담당 직원 알림
        ↓
고객 대응 및 서비스 회복
        ↓
경영진 리포트
```

### 4.1 시스템과 AI의 역할 분리

| 영역 | 담당 |
|---|---|
| 수치 계산 | SQL, Pandas |
| 이상징후 탐지 | 증감률, 이동평균, Z-score, 규칙 |
| 리뷰 감성·주제 분류 | AI 또는 NLP 모델 |
| 자연어 질문 해석 | LLM |
| 분석 결과 설명 | LLM |
| 원인 후보 판정 | 사전 정의된 규칙 |
| 최종 의사결정 | 호텔 운영자 |

## 5. 호텔 VOC에 AI가 본격 적용된 시점

### 5.1 2010년대

- 설문과 리뷰 통합
- 텍스트 분석
- 감성 분석
- 소셜 피드백 분석
- 실시간 직원 알림

이 시기의 핵심은 생성형 AI보다 머신러닝과 규칙 기반 분석이었습니다.

### 5.2 2022년 전후

- 실시간 양방향 메시징
- 고객 요청 자동화
- 현장 직원 알림
- 미처리 문제 에스컬레이션

### 5.3 2024년 이후

- 리뷰 답변 자동 생성
- 고객 피드백 요약
- 자연어 질문
- 개선 우선순위 추천
- 대화형 AI Agent

### 5.4 프로젝트 해석

호텔 VOC 관리에 생성형 AI가 전면에 등장한 시점은 **2024년 전후**로 봅니다.

단, 정확한 최초 도입 시점은 서비스별로 다를 수 있으므로,  
발표나 문서에서 “2024년에 처음 시작됐다”고 단정하지 않습니다.

## 7. 우리가 고도화할 부분

| 기존 서비스 | 우리 서비스 |
|---|---|
| 조식 부정 리뷰 증가 | 조식 불만 증가 시점의 운영 데이터 분석 |
| 부정 VOC +31% | 이용객 +24%, 근무 인원 동일 |
| 대기 관련 불만 반복 | 평균 대기시간 13분 → 25분 |
| 개선 필요 | 혼잡 시간대 인력 배치 우선 점검 |

### 7.1 핵심 고도화 요소

- VOC와 운영지표 교차분석
- 원인 후보 설명
- 근거 리뷰와 수치 동시 제시
- 한국어 호텔 VOC 특화
- 점검 부서와 시간대 제시
- 검토 가능한 개선 옵션 제공

### 7.2 표현 규칙

허용:

> 이용객 증가와 대기시간 증가가 함께 관찰돼 혼잡 시간대 인력 배치를 우선 점검할 수 있습니다.

금지:

> 직원 수 부족 때문에 조식 불만이 증가했습니다.

- Qualtrics 사용 여부
- Medallia 사용 여부

### 8.3 제안 표현

> 공개된 범위에서는 워커힐의 고객용 AI는 확인되지만, 내부 운영자를 위한 VOC·운영 데이터 분석 Agent는 확인되지 않습니다. 따라서 고객용 AI 다음 단계로 내부 운영 의사결정 지원 Agent를 제안할 수 있습니다.

### 8.4 금지 표현

- 워커힐은 해당 시스템을 사용하지 않는다.
- 워커힐은 VOC 분석을 하지 않는다.
- 워커힐 내부 시스템에는 이런 기능이 없다.

8. 규칙 기반 원인 후보 생성
9. 대응 옵션 제시
10. 주간 리포트 생성

### 10.3 예시 결과

```text
조식 관련 부정 VOC가 전주 대비 31% 증가했습니다.

같은 기간 오전 8시부터 9시까지 이용객은 24% 증가했지만,
근무 인원은 동일하게 유지됐습니다.

평균 대기시간은 13분에서 25분으로 증가했습니다.

혼잡 시간대의 인력 배치와 입장 간격을
우선 점검할 필요가 있습니다.
```

### 10.4 검토 가능한 개선 옵션

- 오전 8~9시 추가 인력 배치 검토
- 단체 고객과 일반 고객 이용시간 분산
- 고객별 권장 조식 시간 안내
- 음식 보충 주기 점검

## 11. 활용 데이터

| 데이터 | 주요 컬럼 |
|---|---|
| 고객 리뷰·VOC | 리뷰 일자, 평점, 리뷰 내용, 채널 |
| 시설 운영 | 시설, 시간대, 방문객 수, 수용 인원 |
| 인력 운영 | 시설별 근무 인원, 시간대 |
| 대기시간 | 체크인, 조식, 주차, 부대시설 대기시간 |
| 객실 예약 | 체크인 일자, 투숙 인원, 패키지 |
| 외부 데이터 | 날씨, 주말, 공휴일, 행사 일정 |

### 11.1 데이터 사용 원칙

- 원본 데이터와 정제 데이터를 분리한다.
- 리뷰 원문은 삭제하지 않는다.
- 개인정보는 저장 전 마스킹한다.
- 합성 데이터는 실제 데이터처럼 표현하지 않는다.
- 리뷰와 운영 데이터의 날짜 범위를 일치시킨다.

## 13. 분석 규칙

### 13.1 예시 규칙

```text
조식 부정 VOC 증가율 >= 20%
AND 조식 방문객 증가율 >= 15%
AND 근무 인원 증가율 < 5%
AND 평균 대기시간 증가율 >= 20%
```

### 13.2 출력 예시

> 이용객 증가에 비해 근무 인원이 확대되지 않아 대기시간이 증가했을 가능성을 우선 점검할 필요가 있습니다.

### 13.3 금지 사항

- LLM이 수치를 임의 생성
- 근거 없는 원인 추정
- 상관관계를 인과관계로 표현
- 존재하지 않는 리뷰 원문 생성
- 조회하지 않은 호텔 데이터 사용
