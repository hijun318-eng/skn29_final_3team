## 2. 유지하는 공통 원칙

### 2.1 서비스 경계

- SensePlace는 단일 호텔을 모델링한 합성 VOC·운영 데이터 기반의 내부 운영 지원 PoC다.
- AI는 원인을 확정하거나 조치를 자동 실행하지 않는다.
- 분석 결과에는 관측 사실, 원인 후보, 반대 근거, 부족 데이터를 구분한다.
- 보고서는 근거가 연결된 `DRAFT`로 생성하고 관리자의 승인·보류·반려 이력을 보존한다.

### 2.2 데이터와 재현성

- 실제 고객·직원 식별정보를 생성·수집·저장하지 않는다.
- 합성 데이터에는 `synthetic` 표시, seed, dataset·schema·generator version을 기록한다.
- 원본 리뷰와 학습용 파생 데이터는 분리한다.
- 수치 계산과 이상 판정은 SQL·Python·versioned rule로 수행한다.
- LLM은 허용된 질문 해석과 근거 기반 서술에만 사용하고 실패 시 결정론적 결과를 유지한다.

### 2.3 서비스 책임 경계

- Browser가 호출하는 공개 API의 단일 진입점은 Django다.
- Django는 session·RBAC·scope·job·report decision·감사 이력의 원본이다.
- 장시간 AI 작업은 Django worker와 DB job 상태를 통해 FastAPI에 위임한다.
- FastAPI는 Django가 검증한 scope만 사용하고 허용된 PostgreSQL analysis view를 read-only로 조회한다.
- 구체 endpoint·필드·timeout·retry·library version은 실제 코드와 승인된 구현 설계에서 확정한다.

### 2.4 분석·검색 원칙

- 자연어 조회는 승인된 intent와 `metric_catalog`의 지표·차원·권한 범위로 제한한다.
- SQL은 SELECT-only, allowlist, parameter binding, timeout, row limit을 적용한다.
- 정식 온톨로지·GraphDB를 Baseline 필수 경로로 두지 않는다.
- 장소·VOC 주제·지표 관계는 PostgreSQL 기반 versioned semantic catalog로 먼저 검증한다.
- VectorDB·sLLM·다역할 Agent 비교는 Baseline과 분리된 교육·실험 트랙으로 관리한다.

## 3. 기능 구체화 시 적용할 검토 항목

### 3.1 대화형 분석

- 최초 VOC 질문은 평점 요약을 우선하고, 명시적 추가 질의에서만 허용 기간의 저점 리뷰를 조회한다.
- 결과에는 기간, 단위, role scope, dataset version, 근거 ID를 표시한다.
- 지원하지 않는 질문과 권한 밖 질문은 SQL을 실행하지 않고 거부 사유를 반환한다.

### 3.2 감지·알림·모니터링

- 품질 Gate를 통과한 합성 데이터만 감지 대상으로 사용한다.
- 감지 규칙과 데이터 생성 규칙을 분리하고 서로 다른 담당자가 교차 검토한다.
- 모니터링은 기존 업무 테이블의 조회 view를 사용하며 별도 원천 테이블을 임의로 추가하지 않는다.
- 알림은 담당자 확인·메모·조치 상태를 기록하되 실제 조치를 자동 실행하지 않는다.

### 3.3 Incident·보고서

- Incident에는 관측, 원인 후보, 반대 근거, 부족 데이터, 확인 과제를 함께 둔다.
- 보고서 수치는 결정론적으로 계산하고 LLM 문장에는 evidence ID를 연결한다.
- LLM 실패 시 template 기반 `PARTIAL` 결과를 유지한다.
- 승인된 보고서는 덮어쓰지 않고 새 version이나 사본으로 후속 변경을 관리한다.

## 5. 삭제한 상충 내용

원문 v1.2에서 다음 내용은 활성 기준 문서와 상충하거나 승인 근거가 없어 삭제했다.

- 현행 화면설계서와 다른 `SP-*` 화면 ID·내비게이션·반응형 상세
- 현행 역할 코드와 다른 관리자·운영자·실무자 권한 체계
- Django worker를 생략한 직접 Django→FastAPI 실행 흐름
- 확정되지 않은 frontend·editor·배포 library와 version
- 전량 합성 원칙과 다른 공개 리뷰 corpus 사용 계획
- 정식 온톨로지를 Baseline 공통 계층으로 간주한 설계
- WBS와 다른 구현 단계·완료 판정·비용 개산
- 검증 전 기능을 `충족`으로 표시한 자체 판정

## 6. 검증 기준

- 범위·역할·화면·API를 변경하려면 연결된 요구사항 ID와 WBS ID를 함께 갱신한다.
- 구현 완료·성능·비용 주장은 코드·테스트·로그·승인 기록으로 확인한다.
- 합성 데이터와 목업을 실제 운영 데이터·실시간 연동으로 표현하지 않는다.
- 외부 리뷰·모델·library 정보는 적용 전에 최신 공식 출처와 라이선스를 재검증한다.
