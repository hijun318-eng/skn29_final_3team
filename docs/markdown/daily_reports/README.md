# Codex 보고 갱신 규칙

이 문서는 개인 일일보고, 날짜별 팀 요약과 주간보고의 근거·형식·검증 규칙을 관리한다. Git branch 병합과 commit·push 절차는 `docs/markdown/collaboration/README.md`를 따른다.

## Branch와 보고 파일

| Branch | 보고 파일 |
|---|---|
| `junhee` | `docs/markdown/daily_reports/junhee/일일보고.md` |
| `minji` | `docs/markdown/daily_reports/minji/일일보고.md` |
| `seung` | `docs/markdown/daily_reports/seung/일일보고.md` |
| `daesung` | `docs/markdown/daily_reports/daesung/일일보고.md` |
| `jaehong` | `docs/markdown/daily_reports/jaehong/일일보고.md` |

`main`, `dev` 또는 위 표에 없는 branch에서는 작성자를 임의로 정하지 않고 일일보고를 수정하지 않는다.

## 주차별 팀 요약·주간보고

`docs/markdown/final_project/최종_프로젝트_산출물_및_전체_일정.md`의 `주차 구분`으로 날짜의 `N주차`를 정하고 다음 구조로 저장한다.

```text
team_summaries/
└─ N주차/
   ├─ YYYYMMDD.md
   └─ 주간보고.md
```

- 팀원 5명의 같은 날짜 일일보고는 `team_summaries/N주차/YYYYMMDD.md`에 통합한다.
- 날짜별 파일에는 팀 공통 요약과 팀원별 요약을 두고, 날짜 블록이 없는 팀원은 추정 없이 `보고 없음`으로 표시한다.
- 각 주차 폴더에는 그 주차의 `주간보고.md`를 하나만 두며 다른 주차 내용을 함께 기록하지 않는다.
- 주간보고는 계속 개인 일일보고 5개를 직접 근거로 작성하며 날짜별 팀 요약을 원본 대신 사용하지 않는다.

## 작성 방법

1. 현재 날짜를 KST 기준 `YYYYMMDD`로 구한다.
2. 같은 날짜의 `## YYYYMMDD` 제목이 있으면 그 목록 끝에 수행 내용을 추가한다.
3. 같은 날짜가 없으면 파일 상단 안내문 바로 아래에 새 날짜 블록을 추가한다.
4. 사용자가 이해할 수 있는 완료 결과를 간단한 과거형 또는 명사형으로 기록한다.
5. 기존 보고를 수정하거나 중복 항목을 추가하지 않는다.
6. 제목부터 다음 날짜 제목 직전까지 빈 줄을 포함해 최대 5줄로 작성하고, 초과하면 관련 작업을 한 줄로 통합한다.
7. 개인 branch 작업에서는 개인 일일보고까지만 갱신한다. 팀 요약·주간보고는 사용자가 날짜·기간을 지정해 별도로 요청하거나 아래 `dev 병합 요청 시 보고 통합` 절차가 실행될 때 갱신한다.

```markdown
## YYYYMMDD

- 수행한 작업 또는 완료 결과
```

## 기록 대상

- 코드, 문서, 설정 또는 테스트 파일을 변경한 작업
- 구조 변경, 자동화 추가, 버그 수정 등 저장소에 결과가 남는 작업

다음 작업은 기록하지 않는다.

- 파일 변경이 없는 조사·질문 답변
- commit message 초안 작성처럼 저장소를 변경하지 않은 작업
- 개인 일일보고 누락이나 날짜별 팀 요약·주간보고만 보완한 보고 전용 작업

## dev 병합 요청 시 보고 통합

`docs/markdown/collaboration/README.md`의 절차로 개인 branch를 `dev`에 병합한 직후, push하기 전에 다음 보고 작업을 수행한다.

1. 대상 날짜는 사용자가 지정한 날짜·기간을 우선한다. 지정이 없으면 병합 전후 개인 일일보고에서 추가·변경된 날짜와 개인 보고에는 있지만 날짜별 팀 요약이 아직 없는 날짜를 사용한다.
2. 병합된 `dev`에서 팀원 5명의 개인 일일보고를 읽고 대상 날짜별 `team_summaries/N주차/YYYYMMDD.md`를 갱신한다.
3. 영향받은 각 주차의 `주간보고.md`는 그 주차에 속하면서 개인 보고가 하나 이상 있는 날짜 전체를 원본으로 갱신한다.
4. 공식 주차 매핑, 날짜별 팀원 5명과 `보고 없음` 표시, 개인 일일보고 날짜 블록 5줄, 주간보고 40줄, Git 운영 이력 제외, Markdown 형식과 `git diff --check`를 검증한다.
5. 생성·수정된 `team_summaries/` 경로와 검증 결과를 협업 가이드의 통합 절차에 넘긴다. 보고 통합 자체는 개인 일일보고·팀 요약·주간보고·WBS에 다시 기록하지 않는다.

## 주간보고 작성 규칙

사용자가 날짜 또는 기간과 함께 주간보고 작성을 요청하거나 `dev` 병합 후 보고 통합을 수행할 때 다음 절차로 해당 `team_summaries/N주차/주간보고.md`를 갱신한다. `dev` 병합에서 식별한 날짜는 영향받은 주차를 고르는 데만 사용하고, 각 영향 주차의 보고는 그 주차에 현재 존재하는 개인 일일보고 날짜 전체를 누적 범위로 삼는다.

1. `junhee`, `minji`, `seung`, `daesung`, `jaehong`의 일일보고를 모두 읽는다.
2. `시작일~종료일`이 지정되면 양 끝 날짜를 포함한다.
3. 날짜 하나만 지정되면 해당 날짜를 종료일로 보고, 종료일을 포함한 최근 7일을 대상으로 한다.
4. 공식 일정 문서의 `주차 구분`으로 각 날짜의 `N주차`를 확인하고 해당 주차 폴더를 사용한다.
5. 대상 기간이 여러 주차에 걸치면 하나의 보고서로 합치지 않고 주차별로 나눠 각 `N주차/주간보고.md`를 갱신한다.
6. 사용자가 지정한 기간으로 작성할 때는 해당 주차 안의 지정 날짜만 사용한다. `dev` 병합 후 통합할 때는 영향받은 주차에서 개인 보고가 하나 이상 있는 날짜 전체를 사용한다.
7. 대상 기간의 작업을 기획, 데이터, 모델링, UI/UX, 개발환경, 협업, 멘토링 등 실제 내용에 맞는 상위 작업으로 묶는다.
8. 같은 주차의 보고가 이미 있으면 새 블록을 만들지 않고 기존 파일을 갱신한다.
9. 비슷한 작업은 하나의 상위 항목으로 합치고, 반복되는 설명은 대표 결과 한 번만 남긴다.
10. branch 최신화, fetch·pull·merge, commit·push, commit hash 확인 같은 Git 운영 이력은 제외한다. 해당 이력과 실제 작업 결과가 한 항목에 섞여 있으면 Git 내용만 제거하고 결과는 보존한다.
11. 일일보고에 없는 사실, 완료 여부, 일정 또는 담당자는 추정하지 않는다.
12. 작성자를 사용자가 지정하지 않으면 `3팀`으로 기록한다.
13. 각 `주간보고.md`는 제목과 빈 줄을 포함해 최대 40줄로 제한한다.
14. 40줄을 넘으면 핵심 결과, 결정, 이슈와 다음 작업을 우선해 다시 요약하고 세부 조사 목록과 반복 설명을 줄인다.

주간보고는 다음 형식을 사용한다.

```markdown
# N주차 주간보고

## YYYYMMDD ~ YYYYMMDD (N주차)

작성자: 3팀

- 상위 작업
  - 세부 수행 내용
  - 세부 수행 내용
- 멘토링 진행 (YY.MM.DD HH:mm ~ HH:mm)
  - 피드백 및 후속 작업
```

기간 내 보고가 없는 팀원이 있더라도 빈 항목을 만들지 않는다. 전체 팀원의 보고가 없으면 `- 해당 기간 일일보고 없음`으로 기록한다.
