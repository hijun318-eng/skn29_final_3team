# Codex 일일보고 갱신 규칙

Codex가 저장소 파일을 생성·수정·삭제한 작업을 완료하면 현재 개인 branch의 `일일보고.md`에 실제 수행 결과를 한 줄 이상 기록한다. 새로 추가하거나 수정하는 날짜 블록은 제목과 빈 줄을 포함해 최대 5줄로 제한한다.

## Branch와 보고 파일

| Branch | 보고 파일 |
|---|---|
| `junhee` | `docs/markdown/daily_reports/junhee/일일보고.md` |
| `minji` | `docs/markdown/daily_reports/minji/일일보고.md` |
| `seung` | `docs/markdown/daily_reports/seung/일일보고.md` |
| `daesung` | `docs/markdown/daily_reports/daesung/일일보고.md` |
| `jaehong` | `docs/markdown/daily_reports/jaehong/일일보고.md` |

`main`, `dev` 또는 위 표에 없는 branch에서는 작성자를 임의로 정하지 않고 일일보고를 수정하지 않는다.

## 주차별 팀 요약·주간보고

`docs/markdown/final_project/최종_프로젝트_산출물_및_전체_일정.md`의 `주차 구분`으로 날짜의 `N주차`를 정하고 다음 구조로 저장한다.

```text
team_summaries/
└─ N주차/
   ├─ YYYYMMDD.md
   └─ 주간보고.md
```

- 팀원 5명의 같은 날짜 일일보고는 `team_summaries/N주차/YYYYMMDD.md`에 통합한다.
- 개인 branch에서는 개인 일일보고만 갱신하고, 날짜별 팀 요약과 주간보고의 자동 통합은 해당 branch를 `dev`에 병합한 직후 수행한다.
- 날짜별 파일에는 팀 공통 요약과 팀원별 요약을 두고, 날짜 블록이 없는 팀원은 추정 없이 `보고 없음`으로 표시한다.
- 각 주차 폴더에는 그 주차의 `주간보고.md`를 하나만 두며 다른 주차 내용을 함께 기록하지 않는다.
- branch 최신화, fetch·pull·merge, commit·push와 commit hash 같은 Git 운영 이력은 제외하고 저장소에 남은 작업 결과만 요약한다.
- 주간보고는 계속 개인 일일보고 5개를 직접 근거로 작성하며 날짜별 팀 요약을 원본 대신 사용하지 않는다.
- 날짜별 팀 요약이나 주간보고만 보완한 작업은 개인 일일보고와 다른 보고서에 다시 기록하지 않아 순환 갱신을 막는다.

## 작성 방법

1. 현재 날짜를 KST 기준 `YYYYMMDD`로 구한다.
2. 같은 날짜의 `## YYYYMMDD` 제목이 있으면 그 목록 끝에 수행 내용을 추가한다.
3. 같은 날짜가 없으면 파일 상단 안내문 바로 아래에 새 날짜 블록을 추가한다.
4. 사용자가 이해할 수 있는 완료 결과를 간단한 과거형 또는 명사형으로 기록한다.
5. 기존 보고를 수정하거나 중복 항목을 추가하지 않는다.
6. 제목부터 다음 날짜 제목 직전까지 빈 줄을 포함해 최대 5줄로 작성하고, 초과하면 관련 작업을 한 줄로 통합한다.
7. 개인 branch 작업에서는 개인 일일보고까지만 갱신한다. 팀 요약·주간보고는 사용자가 날짜·기간을 지정해 별도로 요청하거나 아래 `dev 병합 요청 시 보고 통합` 절차가 실행될 때 갱신한다.

```markdown
## YYYYMMDD

- 수행한 작업 또는 완료 결과
```

## 기록 대상

- 코드, 문서, 설정 또는 테스트 파일을 변경한 작업
- 구조 변경, 자동화 추가, 버그 수정 등 저장소에 결과가 남는 작업

다음 작업은 기록하지 않는다.

- 파일 변경이 없는 조사·질문 답변
- commit message 초안 작성처럼 저장소를 변경하지 않은 작업
- 개인 일일보고 누락이나 날짜별 팀 요약·주간보고만 보완한 보고 전용 작업

## dev 병합 요청 시 보고 통합

사용자가 개인 branch의 `dev` 병합을 명시적으로 요청하면 해당 branch를 최신 `dev`에 병합한 직후, `origin/dev`에 push하기 전에 다음 절차를 함께 수행한다.

1. 사용자가 대상 branch를 지정하면 그 branch를 사용한다. 지정하지 않았으면 현재 branch가 `junhee`, `minji`, `seung`, `daesung`, `jaehong` 중 하나일 때만 현재 branch로 정하고, 그 밖의 경우에는 대상을 확인할 때까지 중단한다.
2. 대상 branch의 작업 결과가 모두 commit됐고 작업 트리가 깨끗한지 확인한 뒤 개인 branch를 push한다. `git fetch origin` 후 로컬 대상 branch와 `origin/<대상 branch>`의 commit이 일치하지 않으면 중단한다.
3. `dev`로 전환해 `git fetch origin`과 `git pull --ff-only origin dev`를 실행한 뒤 로컬 `dev`와 `origin/dev`의 commit이 정확히 같은지 확인한다. 로컬 `dev`가 ahead 또는 diverged이면 동일한 미완료 통합을 재개하라는 명시적 요청이 없는 한 중단한다.
4. 병합 직전 `dev` commit을 기록한 뒤 `origin/<대상 branch>`를 병합한다. 기존 미커밋 변경이 있으면 stash·reset·임의 commit하지 않고 중단한다.
5. 병합 충돌이 발생하면 보고 생성과 push를 진행하지 않고 충돌 상태와 파일을 사용자에게 알린다.
6. 대상 날짜는 사용자가 지정한 날짜·기간을 우선한다. 지정이 없으면 병합 전후 개인 일일보고에서 추가·변경된 날짜와 개인 보고에는 있지만 날짜별 팀 요약이 아직 없는 날짜를 사용한다.
7. 병합된 `dev`에서 팀원 5명의 개인 일일보고를 모두 읽고 대상 날짜별 `team_summaries/N주차/YYYYMMDD.md`를 갱신한다. 영향받은 각 주차의 `주간보고.md`는 그 주차에 속하면서 개인 보고가 하나 이상 있는 날짜 전체를 원본으로 갱신한다.
8. 공식 주차 매핑, 날짜별 팀원 5명과 `보고 없음` 표시, 이번 병합에서 추가·수정된 개인 일일보고 날짜 블록 5줄, 주간보고 40줄, Git 운영 이력 제외, Markdown 형식과 `git diff --check`를 검증한다.
9. 이번 절차에서 새로 생성·수정한 `team_summaries/` 파일만 경로를 지정해 stage하고 staged diff를 확인한다. 개인 일일보고·WBS·기능·기획·설정 파일은 stage하지 않는다.
10. staged diff가 있으면 팀 commit 형식의 보고 전용 commit 하나를 만들고, 보고 변경이 없으면 별도 commit을 생략한다.
11. 보고 통합 commit과 병합·push 이력은 개인 일일보고·날짜별 팀 요약·주간보고·WBS에 다시 기록하지 않는다.
12. 모든 검증이 통과한 병합 결과와 보고 전용 commit을 함께 `origin/dev`에 push한다.

`dev` 병합 요청은 대상 개인 branch push, `dev` fetch·pull·merge·push와 위 `team_summaries/` 파일의 stage·commit만 승인한 것으로 본다. 기존 미커밋 변경, 개인 일일보고, WBS와 그 밖의 파일을 새로 stage·commit하는 승인으로 확대하지 않는다. 작업 트리가 깨끗하지 않거나 로컬·원격 commit이 일치하지 않거나 병합 충돌·보고 검증 실패·선행 보고 변경과의 중첩으로 변경을 분리할 수 없으면 중단하고 사용자에게 알린다.

## 주간보고 작성 규칙

사용자가 날짜 또는 기간과 함께 주간보고 작성을 요청하거나 `dev` 병합 후 보고 통합을 수행할 때 다음 절차로 해당 `team_summaries/N주차/주간보고.md`를 갱신한다. `dev` 병합에서 식별한 날짜는 영향받은 주차를 고르는 데만 사용하고, 각 영향 주차의 보고는 그 주차에 현재 존재하는 개인 일일보고 날짜 전체를 누적 범위로 삼는다.

1. `junhee`, `minji`, `seung`, `daesung`, `jaehong`의 일일보고를 모두 읽는다.
2. `시작일~종료일`이 지정되면 양 끝 날짜를 포함한다.
3. 날짜 하나만 지정되면 해당 날짜를 종료일로 보고, 종료일을 포함한 최근 7일을 대상으로 한다.
4. 공식 일정 문서의 `주차 구분`으로 각 날짜의 `N주차`를 확인하고 해당 주차 폴더를 사용한다.
5. 대상 기간이 여러 주차에 걸치면 하나의 보고서로 합치지 않고 주차별로 나눠 각 `N주차/주간보고.md`를 갱신한다.
6. 사용자가 지정한 기간으로 작성할 때는 해당 주차 안의 지정 날짜만 사용한다. `dev` 병합 후 통합할 때는 영향받은 주차에서 개인 보고가 하나 이상 있는 날짜 전체를 사용한다.
7. 대상 기간의 작업을 기획, 데이터, 모델링, UI/UX, 개발환경, 협업, 멘토링 등 실제 내용에 맞는 상위 작업으로 묶는다.
8. 같은 주차의 보고가 이미 있으면 새 블록을 만들지 않고 기존 파일을 갱신한다.
9. 비슷한 작업은 하나의 상위 항목으로 합치고, 반복되는 설명은 대표 결과 한 번만 남긴다.
10. branch 최신화, fetch·pull·merge, commit·push, commit hash 확인 같은 Git 운영 이력은 제외한다. 해당 이력과 실제 작업 결과가 한 항목에 섞여 있으면 Git 내용만 제거하고 결과는 보존한다.
11. 일일보고에 없는 사실, 완료 여부, 일정 또는 담당자는 추정하지 않는다.
12. 작성자를 사용자가 지정하지 않으면 `3팀`으로 기록한다.
13. 각 `주간보고.md`는 제목과 빈 줄을 포함해 최대 40줄로 제한한다.
14. 40줄을 넘으면 핵심 결과, 결정, 이슈와 다음 작업을 우선해 다시 요약하고 세부 조사 목록과 반복 설명을 줄인다.

주간보고는 다음 형식을 사용한다.

```markdown
# N주차 주간보고

## YYYYMMDD ~ YYYYMMDD (N주차)

작성자: 3팀

- 상위 작업
  - 세부 수행 내용
  - 세부 수행 내용
- 멘토링 진행 (YY.MM.DD HH:mm ~ HH:mm)
  - 피드백 및 후속 작업
```

기간 내 보고가 없는 팀원이 있더라도 빈 항목을 만들지 않는다. 전체 팀원의 보고가 없으면 `- 해당 기간 일일보고 없음`으로 기록한다.
