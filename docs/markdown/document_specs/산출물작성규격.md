## 01 요구사항정의서

원본 XLSX의 `분류`는 첫 네 개 데이터 컬럼을 묶는 **병합 표시 헤더**이며 데이터 필드가 아니다.

### 실제 데이터 컬럼

```text
대분류
중분류
요구사항 ID
소분류(기능설명)
추가설명
비고
```

### Markdown 표

```md
| 대분류 | 중분류 | 요구사항 ID | 소분류(기능설명) | 추가설명 | 비고 |
|---|---|---|---|---|---|
| 기능 | VOC 분석 | REQ-F-001 | 리뷰 텍스트의 감성 및 이슈를 분류한다. | 입력·출력 조건 기재 | 필수 |
```

### 작성 기준

- 요구사항 ID는 변경하지 않는 영구 식별자로 사용한다.
- 한 행에는 하나의 검증 가능한 요구사항만 작성한다.
- “사용하기 편해야 한다”처럼 측정 불가능한 표현을 피한다.
- 기능, 비기능, 데이터, 보안, 운영 요구사항은 대분류 또는 중분류로 구분한다.
- 구현 결과 문서와 테스트 문서에서 같은 요구사항 ID를 참조한다.

### 검증 규칙

- 요구사항 ID 중복 금지.
- 요구사항 ID 누락 금지.
- 공식 XLSX의 기본 시트와 필수 필드 구조를 유지한다.
- 우선순위, 상태, 담당자 등 추가 정보는 보조 시트 또는 Markdown의 관련 하위 절에 기록한다. 기본 시트에 추가 컬럼이 필요하면 사용자 또는 팀의 명시적 결정을 받는다.

## 02 WBS

`docs/markdown/02_WBS.md`가 실행 일정·담당·현황·제출일·작업 이력의 단일 기준이다. 공식 공지 변경은 보호된 일정 문서를 수정하지 않고 WBS에 반영한다.

### 02.1 목록형

#### 필수 컬럼

```text
Task ID
주요 업무
세부 업무
담당자
상태
시작일
마감일
우선순위
```

#### Markdown 표

```md
| Task ID | 주요 업무 | 세부 업무 | 담당자 | 상태 | 시작일 | 마감일 | 우선순위 |
|---|---|---|---|---|---|---|---|
| 001 | 기획 | 요구사항 분석 | 홍길동 | 완료 | 2026-07-10 | 2026-07-12 | 높음 |
```

#### 검증 규칙

- `Task ID`는 문서 내에서 고유해야 한다.
- 권장 형식은 3자리 숫자 문자열이다.
- 시작일은 마감일보다 늦을 수 없다.
- 담당자, 상태, 시작일, 마감일은 빈 값 금지.
- 템플릿에 미리 존재하는 행 수나 누락된 예시 번호는 검증 대상이 아니다.
- 공식 XLSX의 기본 시트와 필수 필드 구조를 유지한다.
- 추가 데이터는 별도 보조 시트 또는 관련 Markdown 하위 절에 두며, 기본 시트 구조 변경은 명시적 승인 후 수행한다.

### 02.2 갠트형

#### 프로젝트 메타데이터

```text
프로젝트 이름
프로젝트 관리자
회사명
기준 날짜
```

#### 수동 입력 컬럼

```text
WBS 번호
작업 제목
작업 소유자
시작일
마감일
기간
작업 완료 비율
```

#### 자동 생성 영역

- 1단계~4단계
- 1주~12주
- 월~금 타임라인
- 기간 막대 또는 조건부 서식

#### 검증 규칙

- `기간`은 시작일과 마감일을 기준으로 변환기에서 계산한다.
- `작업 완료 비율`은 0~1 또는 0~100% 범위여야 한다.
- 타임라인 날짜 컬럼은 Markdown 표에 직접 작성하지 않는다.
- 단계·주차·요일 영역은 XLSX 변환기가 생성한다.
- 목록형과 갠트형을 함께 만들 경우 WBS ID, 담당자, 시작일, 마감일이 일치해야 한다.


## 03 프로젝트기획서

### 필수 목차와 순서

```text
1. 프로젝트 주제
2. 문제 정의
3. 시장조사 및 BM 분석
4. 시스템 구성 기획
5. 모델링 계획
6. 데이터 수집 전략
7. 역할분담(R&R)
```

### 작성 기준

| 목차 | 반드시 포함할 내용 |
|---|---|
| 프로젝트 주제 | 프로젝트명, 한 문장 설명, 핵심 사용자, 해결 대상 |
| 문제 정의 | 현재 문제, 원인, 영향, 해결 필요성 |
| 시장조사 및 BM 분석 | 유사 서비스, 차별점, 이해관계자, 활용 가치 |
| 시스템 구성 기획 | 주요 구성요소, 연동 관계, 예상 기술 스택 |
| 모델링 계획 | 모델 또는 에이전트 역할, 학습·평가 계획 |
| 데이터 수집 전략 | 데이터명, 출처, 수집 방법, 법적 검토 계획 |
| 역할분담(R&R) | 팀원, 역할, 담당 업무, 책임 범위 |

### 검증 규칙

- 마지막 페이지의 반복된 “프로젝트 주제” 문구는 템플릿 편집용 자리표시자로 간주하며 목차를 중복 생성하지 않는다.
- 위 7개 필수 목차가 모두 존재하고 상대적 순서가 유지되어야 한다.
- 추가 설명은 가장 가까운 필수 목차의 하위 절로 작성한다.
- 동일 레벨의 별도 목차는 만들지 않으며, 부록도 관련 필수 목차의 하위 절로 배치한다.

## 05 화면설계서

전용 템플릿이 없다. `05_화면설계서_초안.md`를 1차 기준으로 사용하고 `01_요구사항정의서.md`, 실제 구현, `서비스흐름도.png`를 대조한다.

### 권장 필수 내용

```text
1. 화면설계 개요
2. 정보 구조(IA)
3. 사용자 흐름
4. 화면 목록
5. 화면별 구성 요소와 동작
6. 정상·빈 값·로딩·오류·권한 상태
7. 접근성 및 반응형 기준
8. 요구사항 추적표
9. UI QA 결과
```

### 화면 목록 표

```text
화면 ID | 화면명 | 사용자 | 진입 조건 | 주요 기능 | 관련 요구사항 ID
```

### 화면 상세 표

```text
영역 ID | UI 요소 | 입력 / 출력 | 동작 | 상태 / 예외 | 접근성 | 관련 API
```

### 검증 규칙

- 화면 기능은 요구사항 범위를 넘지 않는다.
- 기술명, API, 추적 ID는 실제 구현과 다시 대조한다.
- 개념 흐름도를 기능·데이터 계약으로 사용하지 않는다.
- 추가 화면은 기존 사용자 흐름과 가장 가까운 하위 절에 작성한다.


## 06 데이터베이스저장소설계서

### 필수 목차와 순서

```text
관계형 데이터베이스 (RDB; Relational Database)
1. 모델링 방법
2. 논리 데이터 모델
  2.1 엔터티 목록
  2.2 엔터티 관계
  2.3 ERD
3. 물리 데이터 모델
  3.1 테이블 정의서
  3.2 제약 조건 및 무결성 관리
4. 변경 이력 및 적용 전략
```

### 필수 표

**엔터티 목록**

```text
구분 | 테이블 | 제약 대상 | 제약 유형 | 주요 속성
```

**엔터티 관계**

```text
관계명 | 주 엔터티 | 종 엔터티 | 관계
```

**테이블 정의서**

```text
컬럼명 | 타입 | PK | FK | Not Null | 제약 조건 설명
```

**제약 조건 및 무결성 관리**

```text
적용 레벨 | 제약 유형 | 대상 | 설명 및 관리 방안
```

**변경 이력**

```text
변경일 | 변경자 | 변경내용 | 영향 받는 항목 | 비고
```

### 검증 규칙

- 엔터티 목록의 테이블은 테이블 정의서에 모두 존재해야 한다.
- FK는 엔터티 관계와 일치해야 한다.
- ERD 이미지는 필수이며 상대경로로 연결한다.
- DB에 없는 컬럼을 설명 표에 추가하지 않는다.
- DDL, 인덱스 정의, 운영 정책은 가장 가까운 필수 목차의 하위 절 또는 관련 부록 하위 절로 추가한다.

- 원천 데이터 필드명은 수집 데이터 보고서와 일치해야 한다.

- 학습/검증/테스트 합계가 전체 사용 데이터와 일치해야 한다.
- 랜덤 시드, 버전, 해시 등 재현성 정보가 있어야 한다.
- 전처리 흐름도 필수.
- 전처리 코드와 상세 품질 리포트는 가장 가까운 필수 목차의 하위 절 또는 관련 부록 하위 절로 추가한다.

- 외부 시스템 또는 API
- 요청과 응답 방향
- 비동기 또는 스트리밍 경로
- LangGraph State 흐름

### 검증 규칙

- 그림에 표시된 구성요소가 본문의 기술 스택 표와 일치해야 한다.
- 사용하지 않는 예시 기술을 남기지 않는다.
- 기술 버전을 `최신`으로만 작성하지 않고 실제 버전을 기록한다.
- 보안 체크는 구현 증거가 있을 때만 완료로 표시한다.

## 17 개발된LLM연동웹애플리케이션

### 필수 목차와 순서

```text
3. 보안 정보 및 코드 관리 평가
4. 예외 처리 및 서비스 완성도 확인
5. 종합 판정, 강점, 한계 및 개선 계획
```

### 필수 표

```text
ID | 기능 요구사항 | 구현 내용 | 충족 여부(P/F)
```

```text

```

### 검증 규칙

- `ID`는 요구사항 정의서의 요구사항 ID를 그대로 사용한다.
- 구현 내용에는 실제 화면, API, 코드 경로 또는 테스트 근거를 연결한다.
- 비밀키 값을 문서에 직접 적지 않는다.
- `.env`, Secret Manager, `.gitignore`, PII 필터링 여부를 검사한다.
- 종합 판정은 개별 요구사항과 테스트 결과에 근거해야 한다.

- 코드와 문서의 명령어, 포트, 버전, 파일 경로가 일치해야 한다.
