## 3팀 프로젝트 정의

사용자가 자연어로 질문하면 AI가 질문의 의도를 파악하고 데이터를 조회·분석한 뒤, 데이터에 근거한 해석과 리포트를 제공하는 서비스다.

예를 들어 사용자가 “이번 달 매출이 가장 많이 오른 상품은 뭐야?”라고 물으면 다음과 같이 동작한다.

1. 데이터 이해 에이전트가 질문 의도를 파악한다.
2. SQL 또는 분석 에이전트가 필요한 데이터를 조회·분석한다.
3. 리포트 작성 에이전트가 결과를 읽기 쉬운 문서 형태로 정리한다.

핵심은 단순 질의응답이 아니라 **자연어 질문 → 근거 데이터 조회 → 분석 → 해석 및 리포트 자동 생성**의 전체 흐름을 구현하는 것이다.

### 제시된 활용 예시

- 고객 VOC·리뷰 분석 리포팅: 긍정/부정/불만 유형 분류 및 월간 개선 리포트 생성
- 지점·서비스 운영 KPI 모니터링: 매출, 방문 수, 재방문율, CS 이슈 비교와 이상 징후 탐지
- 호텔·리조트 수요 예측 및 운영 리포트: 시즌·지역·이벤트 기반 수요 예측, 인력·프로모션 운영 인사이트 제공
- ESG·지속가능경영 대시보드: 에너지 사용, 폐기물, 친환경 운영 지표 시각화 및 개선 과제 추천

## 3팀 필수 수행 단계와 산출 방향

프로젝트는 아래 4단계를 모두 수행해야 하며, 단계별 산출물을 강사와 멘토가 평가한다.

### 1. 데이터 수집 및 저장

- 대화형 데이터 서비스 구현에 필요한 데이터 수집
- 관계형 DB 또는 분석용 저장 구조 설계·구축
- 데이터 출처, 스키마, 품질과 사용 근거 문서화

### 2. 데이터 전처리

- 데이터 이상치 탐지, 수요 예측, 데이터 분류·군집화에 사용할 머신러닝/딥러닝 모델 선정 및 테스트
- 학습·평가 데이터 분리와 평가 지표 명시

### 3. 모델링 및 평가

- LangChain 또는 LangGraph 기반 RAG 구성
- 질의 해석, 데이터 분석, 시각화, 리포트 생성을 담당하는 멀티 에이전트 구현
- 테스트, 평가, 오류 분석 및 개선
- 필요 시 파인튜닝을 통한 자체 LLM 개발
- 산출물 안내에서는 3·4번 주제에 자체 LLM 및 sLLM 파인튜닝 평가 결과를 포함하도록 표시되어 있으므로 필수 여부를 담당자에게 확인

### 4. 모델 배포

- 자연어 기반 데이터 분석 결과와 자동 보고서를 제공하는 웹 플랫폼 개발
- 자연어 질의, 데이터 조회·분석, 근거 제시, 리포트 생성까지 하나의 사용자 흐름으로 연결
- 실제 시연 가능한 상태로 배포

## 권장 시스템 흐름

```text
사용자 자연어 질문
        ↓
질문 의도·업무 유형 분석
        ↓
SQL 생성 또는 데이터 분석 에이전트
        ↓
DB 조회 + 분석/예측
        ↓
결과 검증 및 근거 연결
        ↓
해석·차트·리포트 자동 생성
        ↓
웹 화면 제공 및 리포트 저장/내보내기
```

초기 기획에서는 반드시 다음을 명확히 해야 한다.

- 누가 사용하는가?
- 어떤 문제를 해결하는가?
- SK네트웍스 그룹사의 어떤 비즈니스 맥락에서 활용되는가?

### 주제·기획 공통 제약

- 실제 그룹사 내부 데이터가 아니라 **공공데이터·오픈데이터** 등을 활용해 현업 적용 가능한 AI 서비스를 기획·구현한다.
- 비즈니스 분석을 통해 해결할 문제를 정의하고, 그룹사 내 활용 맥락과 필요한 데이터를 탐색한 뒤 멀티 에이전트를 설계한다.
- 발표 자료에 **비즈니스 연계성 설명을 1페이지 분량으로 반드시 포함**한다.
- 기술 구현뿐 아니라 핵심 고객, 현재 운영 문제, 비용 절감·응대 품질·운영 효율과의 연결, 기술 선택과 비즈니스 목적의 정합성을 설명해야 한다.
- 정해진 대주제 안에서 대상 그룹사, 세부 서비스 시나리오, 사용자, 데이터 수집 방식과 구현 방식은 팀이 자율적으로 설계한다.
- 확정된 팀 주제는 임의 변경할 수 없다.
- 강사와 멘토는 기술·프로젝트 진행을 지도하지만, 프로젝트의 주도권과 주요 의사결정 책임은 팀에 있다.

## 공통 산출물

자료에서 요구하는 산출물의 큰 범주는 다음과 같다.

- 기획·설계 자료: 서비스 정의, 사용자와 문제, 요구사항, 데이터 및 시스템 설계
- 데이터 관련 자료: 수집·정제 과정, DB/GraphDB 설계, 분석 또는 ML/DL 결과
- AI 관련 자료: RAG/멀티 에이전트 구성, LLM 평가, 필요 시 sLLM 파인튜닝 결과
- 구현 결과: 웹서비스, 배포 환경, 시연 가능한 데모
- 발표 자료: 중간·최종 발표 PPT와 최종 결과 정리

세부 양식과 제출 위치는 공지된 공유 드라이브 및 담당자 안내를 따라야 한다. 단계별 피드백을 반영하고, 수정 이력을 관리하는 것이 중요하다.

### PDF에 명시된 세부 산출물

- 데이터 수집 및 저장: 데이터베이스/저장소 설계 문서, 수집 데이터 보고서
- 데이터 전처리: 데이터 전처리 결과서, 머신러닝/딥러닝 학습결과서, 학습한 ML/DL 모델
- 모델링 및 평가: 벡터DB/GraphDB 구축 결과서, AI 시스템 아키텍처, 멀티 에이전트 테스트 계획 및 결과 보고서와 소스코드, 자체 LLM 인공지능
- 모델 배포: 요구사항 정의서, 화면설계서, 개발된 LLM 연동 웹 애플리케이션, 시스템 구성도, 테스트 계획 및 결과 보고서
- 발표 및 시연: 중간 발표 PPT, 최종 발표 PPT, 프로젝트 개발 소스코드, 시연영상

정확한 마감일과 작성 요구사항은 별도 문서인 `최종_프로젝트_산출물_및_전체_일정.md`를 기준으로 관리한다.

## 팀 운영 그라운드 룰

- 데일리 스크럼은 **월~목요일 8교시**에 진행하며, 5분 안에 팀원 1명씩 순환 발표한다. 구현 기능, 해결한 문제·트러블슈팅, AI 활용 사례, Git 협업 경험, 다음 날 업무 등 자유 주제로 공유한다.
- 주간 스탠드업은 **매주 금요일 8교시**에 진행한다. 팀 전체 발표 시간은 5~10분이며 주차별 진행 상황, 이슈와 차주 계획을 공유한다.
- 주간 업무 일지는 **목요일 밤까지 필수 작성**하며 PM 또는 팀원이 조율해 작성할 수 있다. 금요일 보고자는 반드시 참석한다.
- 강사 정기 미팅은 팀별 지정 요일에 진행하며 **3팀은 매주 수요일**이다. 팀과 강사가 조율해 일정을 변경할 수 있으며, 해당 팀은 반드시 참석한다.
- PM과 별도로 `파랑새` 역할을 지정한다. 파랑새는 주간 보고 일지 작성 관리, 멘토링 일정 조율과 진행 현황 보고, 산출물 일정·제출 리마인드, 팀 이슈 공유와 리소스 관리를 지원한다.
- 파랑새 소통 채널에서 담당 매니저와 밀접하게 소통하고, 멘토링 후 진행 현황을 공유한다.
- 이슈 발생 시 원인과 대안을 함께 제시하고, 담당 범위를 넘어 팀 목표 달성에 필요한 일을 지원한다.

- [ ] 정확도, SQL 실행 성공률, 근거 일치율, 응답 시간 등 평가 지표 선정

## 가장 중요한 성공 기준

3팀은 기능 수를 늘리기보다 다음 연결을 완성도 높게 증명하는 것이 중요하다.

> **실제 비즈니스 질문 → 정확한 데이터 조회·분석 → 검증 가능한 근거 → 의사결정에 유용한 해석·리포트 → 배포된 웹서비스**

데모에서는 답변이 그럴듯한지를 넘어, 어떤 데이터와 계산을 근거로 결론을 냈는지 사용자가 확인할 수 있어야 한다.
