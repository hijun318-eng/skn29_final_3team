# 문서 관리 규칙

## 기본 구조

`docs/` 바로 아래에는 다음 세 폴더, 이 규칙 문서, 그리고 소수의 보조 파일만 둔다.

```text
docs/
├─ 문서관리규칙.md
├─ 서비스흐름도.png 등 보조 파일
├─ markdown/
│  ├─ 01_요구사항정의서.md 등 번호가 붙은 산출물 작업본
│  ├─ walkerhill/
│  ├─ voc/
│  ├─ final_project/
│  ├─ collaboration/
│  └─ daily_reports/
│     └─ team_summaries/
│        └─ N주차/
│           ├─ YYYYMMDD.md
│           └─ 주간보고.md
├─ deliverables/
└─ templates/
```

- `markdown/`: `docs/`에서 관리하는 모든 Markdown 문서와 기존 협업·보고 하위 폴더
- `deliverables/`: 우리 팀이 제출하는 공식 산출물 문서, 이미지, 영상, 모델 및 소스 묶음. 제출물 외의 파일은 두지 않는다.
- `templates/`: 제공받은 읽기 전용 원본 양식. 원본 파일명과 내용을 유지하며, 제출물과 섞이지 않도록 `deliverables/` 밖에 둔다.
- `문서관리규칙.md`: 이 규칙 문서. `docs/` 바로 아래에 둔다.
- 보조 파일: 공식 산출물이 아닌 이미지·다이어그램 등(`서비스흐름도.png` 등)은 `docs/` 바로 아래에 둘 수 있다. 수가 많아지면 성격에 맞는 하위 폴더로 정리한다.

### 읽기 전용 보호 폴더

- `docs/markdown/final_project/`와 `docs/templates/`의 모든 파일은 기준 자료로만 읽는다.
- 두 폴더에서는 파일 생성·수정·삭제·이동·이름 변경·덮어쓰기를 금지하며, 자동 formatting·일괄 치환·링크 경로 갱신도 수행하지 않는다.
- 공식 일정이나 원본 양식의 변경 필요성을 발견하면 보호 파일을 수정하지 않고 사용자에게 알린다. 후속 내용은 `docs/markdown/final_project/` 밖의 작업 문서, `docs/markdown/02_WBS.md` 또는 `docs/deliverables/`에 목적에 맞게 반영한다.

## 답변 참고 라우팅

아래 규칙은 사용자 질문에 답할 때 기존 Markdown 문서를 찾는 기준이다. 새 문서 생성이나 답변 저장 위치를 정하는 규칙이 아니다. 저장소 문서로 답변할 수 있으면 해당 문서를 먼저 읽고, 문서에 없는 내용이나 시점에 따라 바뀔 수 있는 정보는 확인 한계를 밝히고 필요한 경우 최신 공식 출처로 재검증한다.

| 질문의 중심 주제 | 우선 확인 위치 | 답변 기준 |
|---|---|---|
| 워커힐 기업·브랜드·시설·시장·비즈니스 모델·AI Guide·WISE·공식 VOC 채널·워커힐 적용안 | `docs/markdown/walkerhill/`의 관련 Markdown 전체 | `워커힐_기업분석.md`를 종합 기준으로 삼고 세부 주제 문서를 함께 확인한다. 워커힐의 변동 가능한 사실은 문서 기준일과 최신 공식 출처를 대조한다. |
| 범용 호텔 VOC, VOC-to-Action, AI Agent의 목표·기능·데이터·분석·평가·안전·Human-in-the-loop | `docs/markdown/voc/`의 관련 Markdown 전체 | `HOTEL_VOC_AI_AGENT.md`를 서비스 정의 기준으로 삼고 `VOC_핵심개선사항_분석.md`를 방법론·고도화 참고로 사용한다. 프로젝트 범위 질문이면 `01_요구사항정의서.md`와 `02_WBS.md`도 확인한다. |
| 프로젝트 요구사항·일정·기획·화면 | `docs/markdown/`의 번호 문서 | 아래 `작업별 참고 문서` 표에서 질문에 맞는 1차 기준 문서를 선택한다. |

| Git 협업 또는 일일·주간보고 | `docs/markdown/collaboration/`, `docs/markdown/daily_reports/` | 각 폴더의 `README.md`를 먼저 읽고 실제 branch 상태나 개별 보고 내용을 함께 확인한다. |
| 위 분류에 없는 질문 | `docs/markdown/` 전체와 아래 작업별 표 | 파일명만으로 판단하지 않고 관련 문서의 범위·기준일·근거 수준을 확인한다. 문서 근거가 부족하면 추정하지 않는다. |

혼합 질문은 관련 폴더를 모두 확인한다. 예를 들어 “워커힐에 VOC Agent를 적용하는 방법”은 `walkerhill/`을 1차, `voc/`를 보조 근거로 사용하고, “호텔 VOC Agent를 설계하면서 워커힐을 사례로 사용”은 `voc/`를 1차, `walkerhill/`을 사례 근거로 사용한다.

### 참고 우선순위와 충돌 처리

1. **작업 절차:** 사용자의 현재 명시 지시 → 저장소 루트 `AGENTS.md` → 이 문서 → 작업별 전용 README 순서로 적용한다.
2. **내용 판단:** 최신 공식 공지·실제 코드·테스트·데이터·제출본 → 위 표의 1차 기준 문서 → 보조 분석 문서 → 일일·주간보고 → `templates/` 순서로 확인한다.
3. 요구사항·범위·수용 기준은 `01_요구사항정의서.md`, 실행 일정·담당·현황은 `02_WBS.md`, 화면 상세는 요구사항을 준수하는 범위에서 `05_화면설계서_초안.md`, 공식 마감·번호는 최종 프로젝트 전체 일정 문서를 우선한다.
4. 워커힐의 최신 상태·시장 수치·AI Guide·WISE 등 변동 가능한 사실은 문서 기준일을 확인하고 공식 출처로 재검증한다.
5. 문서의 `[사실]`과 직접 근거는 `[해석]`, `[가정]`, `[추가 제안]`보다 우선한다. `현재 구현` 주장은 문서만으로 확정하지 않고 실제 코드·WBS·테스트를 확인한다.
6. 충돌을 발견하면 임의로 내용을 합치지 말고 차이, 기준일과 우선 기준을 기록한 뒤 필요한 확인 사항을 남긴다.

### 템플릿과 제출본 사용

- `docs/templates/`의 18개 파일은 제공받은 27기 읽기 전용 원본 양식이다. 파일명과 원본 내용을 변경하지 않으며, 아래 번호별 매핑이 있는 산출물에는 목차·구조와 필수 필드의 기준으로만 사용한다.
- 파일명이 정규식 `^\d{2}_`에 해당하는 Markdown 작업본과 공식 제출 문서는 편집 전에 대응 양식을 직접 확인하고, 양식 목차의 제목·순서·계층을 유지한다. 프로젝트 고유 설명·검증·부록은 가장 가까운 양식 항목의 하위 절로만 추가하고, 같은 레벨의 별도 목차를 임의로 만들지 않는다.
- DOCX에 명시적 목차가 없으면 본문 섹션 순서를 기준으로 한다. XLSX는 시트·필드 구조, PPTX는 슬라이드 섹션 순서를 대응 구조로 본다.
- 대응 양식이 둘 이상이거나 전용 양식이 없거나 매핑이 불명확하면 임의 양식을 공식 기준으로 지정하지 않는다. 아래 매핑표와 최종 프로젝트 전체 일정의 작성 요건을 따르고, 결과를 크게 바꾸는 선택은 사용자 또는 팀의 결정을 받는다.
- 기존 번호 문서도 다음 내용 편집부터 이 규칙을 적용한다. 명시적 요청 없이 다른 번호 문서를 일괄 재구성하지 않는다.
- 현재 전용 템플릿이 없는 산출물은 `05 화면설계서`, `13 LLM 활용 소프트웨어`, `20 프로젝트 개발 소스코드`, `21 시연영상`이다. 임의 양식을 공식 양식으로 표현하지 않고 전체 일정 문서의 작성 요건과 관련 프로젝트 문서를 따른다.
- `docs/deliverables/`의 `01_요구사항정의서_29기_3팀.xlsx`, `02_WBS_29기_3팀.xlsx`는 현재 공식 제출 파일이다. 같은 번호 Markdown을 먼저 갱신한 뒤 제출 전 동기화 여부를 검증한다.

## 파일 이름 규칙

문서를 세 부류로 나누고 부류별 이름 형식을 따른다.

### 1. 공식 산출물 — `docs/deliverables/`

`두 자리 번호_문서이름_29기_3팀.확장자` 형식을 사용한다.

예시:

```text
01_요구사항정의서_29기_3팀.xlsx
02_WBS_29기_3팀.xlsx
03_프로젝트기획서_29기_3팀.docx
```

- 번호는 `docs/markdown/final_project/최종_프로젝트_산출물_및_전체_일정.md`의 마감일 순서를 따르며 아래 산출물 번호 표와 같다.
- 여러 파일이 필요한 산출물은 같은 이름의 폴더를 만든다.

```text
docs/deliverables/10_학습MLDL모델_29기_3팀/
docs/deliverables/17_개발된LLM연동웹애플리케이션_29기_3팀/
```

### 2. 산출물 연관 작업 문서 — `docs/markdown/`

공식 산출물의 원고나 작업본이 되는 Markdown 문서는 산출물 번호를 앞에 붙인다.

- 팀 공동 작업본: `두 자리 번호_산출물이름.md`
- 개인 작업본·초안: `두 자리 번호_산출물이름_구분어.md` (구분어는 작성자 이름이나 `초안` 같은 상태 표기)

예시:

```text
01_요구사항정의서.md
02_WBS.md
05_화면설계서_초안.md
```

- 번호와 산출물 이름은 아래 산출물 번호 표를 그대로 사용하고, 임의로 새 번호나 다른 이름을 만들지 않는다.
- 같은 번호의 문서가 곧 `deliverables/` 제출 파일의 원본이 된다.

### 3. 일반 문서 — `docs/markdown/`

산출물과 직접 연관되지 않는 조사·분석·기록 문서는 번호를 붙이지 않는다. 고정 형식은 없지만, 사람과 AI가 파일명만 보고 문서의 주제와 성격을 판단할 수 있도록 다음을 지킨다.

- `대상_내용.md` 순서로 큰 주제를 앞에 둔다. 예: `워커힐_기업분석.md`, `워커힐_호텔_BM_분석.md`
- 파일명에 문서 성격(분석, 조사, 요약, 정리, 계획 등)을 나타내는 단어를 포함한다.
- 약어보다 뜻이 드러나는 완전한 단어를 쓰고, 공백 대신 밑줄(`_`)을 쓴다.
- 날짜가 필요한 문서는 `YYYYMMDD` 형식을 파일명 끝에 붙인다.
- 개인 일일보고는 기존 branch별 구조를 유지하고, 날짜별 팀 요약과 주간보고는 각각 `daily_reports/team_summaries/N주차/YYYYMMDD.md`, `daily_reports/team_summaries/N주차/주간보고.md` 형식으로 저장한다.
- 같은 주제의 일반 문서가 여러 개이면 `walkerhill/`, `voc/`, `final_project/`처럼 소문자 영문 하위 폴더로 묶는다.

### 공통

- 개인 작업본과 조사 문서는 공식 제출본으로 오해되지 않도록 산출물 이름 규칙(1번 형식)을 쓰지 않는다.
- 작업 중 버전은 필요할 때만 파일명 끝에 `_v01`, `_v02`를 붙이고 최종 제출본에서는 제거한다.

## 문서 헤더 규칙

모든 Markdown 문서는 최상위 제목(`#`) 바로 아래에 메타데이터 표를 둔다. 표는 미리보기에서 표 형태로 렌더링되도록 `| 항목 | 내용 |` 형식을 쓰고, 문서 부류에 따라 아래 행을 가감한다. `docs/markdown/final_project/`와 `docs/templates/` 보호 폴더의 파일은 이 규칙 대상이 아니며 기존 내용을 수정하지 않는다.

### 공통 최소 헤더

모든 문서가 공통으로 포함하는 최소 행이다. 행 이름과 순서를 유지한다.

```markdown
# {문서 제목}

| 항목 | 내용 |
|---|---|
| 문서 설명 | 호텔 VOC·운영 지원 플랫폼 기획 근거를 정리한 작업본 |
| 문서 분류 | 산출물 작업본 |
| 버전 | v1.0 |
| 문서 기준일 | 2026-07-21 14:30 |
| 작성·수정 | 윤대성 |
```

- **문서 설명**: 이 문서가 무엇인지 한 줄로 요약한다. 파일명만으로 알기 어려운 목적·범위를 담는다.
- **문서 분류**: `산출물 작업본`, `일반 문서`, `일일·주간보고` 중 하나. 파일 이름 규칙의 부류와 일치시킨다.
- **버전**: `vX.Y`. 내용이 바뀌면 올린다. 파생 요약 문서는 `—`로 둘 수 있다.
- **문서 기준일**: 마지막으로 내용을 갱신·검증한 시점. 날짜와 시간을 함께 적는다(`YYYY-MM-DD HH:MM`, KST).
- **작성·수정**: 문서를 실제로 작성·수정한 사람의 이름. **필수 항목이며 `—`나 공란으로 두지 않는다.**
  - 편집한 사람은 자기 이름을 그대로 적는다. **자신이 편집했으면서 다른 사람 이름이나 가짜 이름을 적어 실제 편집자를 숨기는 것을 금지한다.**
  - 공동 문서는 실제로 편집한 사람을 대표 1인 또는 쉼표로 나열한다.

### 부류별 추가 행

부류마다 공통 헤더 표에 아래 행을 이어 붙인다.

**산출물 작업본** (`두 자리 번호_...md`)

| 항목 | 내용 |
|---|---|
| 산출물 번호 | 03 |
| 제출 일자 | 2026-07-24 |
| 대응 템플릿 | `templates/[기획] 프로젝트 기획서_양식_27기_0팀.docx` |

- 산출물 번호와 제출 일자는 위 `산출물 번호` 표와 `markdown/final_project/최종_프로젝트_산출물_및_전체_일정.md`를 따른다.
- 전용 템플릿이 없는 산출물(05·13·20·21)은 `대응 템플릿`을 `없음`으로 적는다.

**일반 문서** (조사·분석·기록) — 추가 행 없이 공통 최소 헤더만 사용한다.

**일일·주간보고** — 무겁게 관리하지 않는다. `버전`은 생략할 수 있고, 다음 행으로 공통 헤더를 대신한다.

| 항목 | 내용 |
|---|---|
| 문서 설명 | 팀원 5명 일일보고를 종합한 날짜별 요약 |
| 문서 분류 | 일일·주간보고 |
| 날짜 | 2026-07-21 |
| 작성 | 윤대성 (`daesung`) |
| 근거 | 팀원 5명 개인 일일보고 `20260721` 블록 |

### 변경 내역 (문서 하단)

문서 맨 아래에 최근 변경만 간단히 요약한다. 상세 이력은 Git이 관리하므로 최근 3~5건만 남기고, 한 줄 요약으로 짧게 적는다.

```markdown
## 변경 내역

| 버전 | 일시 | 요약 |
|---|---|---|
| v1.1 | 2026-07-21 14:30 | 근거 기준일·시장 수치 갱신 |
| v1.0 | 2026-07-20 09:00 | 최초 작성 |
```

- 일시는 헤더의 `문서 기준일`과 같은 `YYYY-MM-DD HH:MM` 형식을 쓴다.
- 새 항목은 맨 위에 추가하고, 오래된 항목은 3~5건을 넘기면 지운다.
- 일일·주간보고 등 매번 새로 쓰는 짧은 문서에는 변경 내역을 두지 않아도 된다.

### 헤더 작성 원칙

- 행 이름과 순서를 임의로 바꾸지 않는다. 해당 없는 행은 삭제하지 말고 값에 `—`를 넣는다. 단 `작성·수정`은 예외로, 언제나 실제 편집자 이름을 채운다.
- `작성·수정`에는 실제로 편집한 사람의 이름만 적는다. 다른 사람 이름이나 가짜 이름으로 바꿔 적어 편집자를 숨기지 않는다.
- 문서 고유 행(프로젝트 기간, 대상 사용자 등)이 필요하면 표준 행 아래에 더 추가할 수 있다. 단 공통 최소 행은 반드시 포함한다.
- 날짜·시간은 `YYYY-MM-DD HH:MM`, 버전은 `vX.Y` 형식을 지킨다.
- `버전`·`문서 기준일`은 내용을 편집할 때마다 갱신하고, 하단 변경 내역에 한 줄을 더한다.
- 기존 문서는 다음 내용 편집부터 이 헤더를 적용한다. 명시적 요청 없이 전체 문서를 일괄 변경하지 않는다.

## 변경 시 점검

1. 파일 종류에 맞는 폴더인지 확인한다.
2. 공식 산출물이거나 산출물 연관 작업 문서이면 일정 문서에서 번호를 확인한다.
3. 파일명이 `두 자리 번호_`로 시작하면 대응 템플릿을 직접 확인하고 최상위 제목·순서·계층 및 필수 필드를 대조한다.
4. 기존 파일을 이동하거나 이름을 바꾸면 관련 Markdown 링크와 프로젝트 지침(`AGENTS.md` 포함)의 경로도 갱신한다.
5. `docs/` 바로 아래에 임시 폴더를 만들지 않는다. 보조 파일은 공식 산출물이 아닌 것만 두고, 늘어나면 하위 폴더로 정리한다.
6. 새 문서를 만들기 전에 이 규칙에서 어느 부류에 속하는지 판단하고, 애매하면 일반 문서(3번)로 두었다가 산출물로 확정될 때 번호를 붙인다.
7. 문서를 만들거나 편집하면 `문서 헤더 규칙`에 따라 상단 헤더 표를 두고 `버전`·`문서 기준일`을 갱신한 뒤 하단 변경 내역에 한 줄을 더한다.
