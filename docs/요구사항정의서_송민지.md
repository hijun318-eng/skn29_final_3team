# 워커힐 VOC·운영 이슈 분석 AI Agent 요구사항 정의서

> 엑셀 요구사항 정의서의 내용을 표 없이 읽을 수 있도록 문장 중심으로 재구성한 문서입니다.
> 시설·주제·운영지표 매핑과 실제 연동 방식은 고객사 협의 후 확정합니다.

## 1. 프로젝트 요약

- **프로젝트 목적:** 워커힐의 VOC와 운영지표를 동일 시설·동일 기간 기준으로 교차분석하여 원인 후보, 근거 데이터, 우선 점검 항목과 검토 가능한 대응 옵션을 내부 운영자에게 제공한다.
- **주요 사용자:** 총지배인·경영진, 고객경험 담당자, F&B 팀장, 객실 운영팀장, 프런트·주차·시설 운영 담당자, 시스템 관리자
- **Core MVP 범위:** CSV 기반 VOC·합성 운영 데이터 업로드, 전처리, 감성·다중 주제 분류, VOC·운영 KPI 계산, 규칙 기반 이상징후 및 교차분석, 사전 정의 질의 10개, 실무자용 대시보드, 근거 리뷰, 주간 HTML 리포트
- **Enhanced MVP 범위:** 데모 계정 로그인, 역할별 화면, 설정 관리, 분석·리포트 이력, 감사 로그, 다운로드 통제와 실패 작업 재처리는 Core MVP 완료 후 일정과 배포 방식에 따라 구현한다.
- **주요 가정:** MVP는 공개 리뷰와 프로젝트용 합성 운영 데이터를 사용한다. 시설·주제·운영지표 매핑은 제안 기준이며 고객사 협의 후 확정한다.
- **미확정 사항:** 실제 데이터 구조·보유 기간·건수, PMS·POS·CRM·VOC·설문·인력관리 시스템 종류와 연동 방식, 사내 인증 방식, 실제 권한 체계는 확인되지 않았다.
- **핵심 통제:** 계산은 SQL·Pandas·통계 규칙으로 수행하고 LLM은 분류·질의 해석·설명 생성에 사용한다. 상관관계를 인과관계로 표현하지 않으며 최종 판단과 조치는 운영자가 수행한다.
- **문서 형식:** Git 저장소의 `[모델배포]_요구사항 정의서_양식_27기_0팀.xlsx`에 맞춰 `분류 / 대분류 / 중분류 / 요구사항 ID / 소분류(기능설명) / 추가설명 / 비고` 구조를 유지한다.
- **요구사항 수:** 총 68개: Core MVP 38개, Enhanced MVP 21개, 차기·외부 연동 9개

### 2.1 구현 우선순위

- **Core MVP 38개:** `BIZ-001`, `BIZ-002`, `BIZ-004`, `FUN-003`~`FUN-007`, `DAT-001`~`DAT-007`, `AI-001`, `AI-002`, `AI-004`~`AI-009`, `UI-001`~`UI-003`, `UI-005`, `RPT-001`, `SEC-002`, `SEC-003`, `NFR-004`, `NFR-005`, `OPS-002`, `OPS-003`, `TST-001`~`TST-003`, `TST-005`
- **Enhanced MVP 21개:** `BIZ-003`, `FUN-001`, `FUN-002`, `FUN-008`~`FUN-010`, `AI-003`, `UI-004`, `RPT-002`, `RPT-004`, `SEC-001`, `SEC-004`, `SEC-005`, `NFR-001`~`NFR-003`, `NFR-006`, `OPS-001`, `OPS-004`, `OPS-005`, `TST-004`
- **차기·외부 연동 9개:** `RPT-003`, `INT-001`~`INT-004`, `FUT-001`~`FUT-004`
- 우선순위의 `Must`는 해당 기능을 구현할 때 반드시 만족해야 하는 품질 수준을 뜻한다. 실제 구현 순서는 위의 Core·Enhanced·차기 범위를 우선 적용한다.

### 2.2 MVP v1 데이터 계약

- VOC 입력은 UTF-8 CSV를 사용하며 필수 컬럼은 `review_id`, `hotel_id`, `source`, `review_date`, `rating_original`, `review_text_original`, `channel`, `facility`이다.
- `review_id`는 문자열 고유값, 날짜는 `YYYY-MM-DD`, 평점은 원본 척도와 정규화 점수를 함께 보존한다.
- 운영 데이터는 `hotel_id`, `facility`, `date`, `hour`, `visitor_count`, `staff_count`, `wait_time_minutes`, `capacity`, `occupancy_rate`, `group_guest_count`, `event_flag`, `holiday_flag`를 사용한다.
- 성능 검증 기준 데이터는 VOC 최대 10,000건과 30일치 시설별 시간 단위 운영 데이터로 고정한다.
- 합성 운영 데이터에는 `synthetic=true`, `seed=29`, `schema_version=mvp_v1`을 기록한다. seed나 schema가 바뀌면 별도 데이터 버전으로 관리한다.
- 공개 리뷰는 출처 URL, 수집일, 이용 조건을 기록하고 실제 고객사 데이터와 혼합하지 않는다.

### 2.3 KPI·이상징후 계산 기준

- 비교 기간은 사용자가 선택한 현재 기간과 바로 이전의 동일 길이 기간으로 정의한다.
- 부정 VOC 비율은 `부정 리뷰 수 / 감성 분류가 완료된 전체 리뷰 수`로 계산하고, 기간 간 변화는 퍼센트포인트 차이와 건수 증감률을 함께 제공한다.
- 건수 증감률은 `(현재 건수 - 이전 건수) / max(이전 건수, 1) × 100`으로 계산한다.
- 평균 대기시간, 방문객 수, 근무 인원과 점유율은 동일 시설·동일 시간 단위로 집계한다.
- MVP 이상징후는 현재 기간의 유효 리뷰가 10건 이상이면서 `부정 VOC 건수 20% 이상 증가` 또는 `부정 VOC 비율 10%p 이상 증가`인 경우 후보로 생성한다.
- 원인 후보는 VOC 이상징후와 함께 방문객·대기시간·근무 인원 등 관련 운영지표가 15% 이상 변한 경우에만 제시한다. 임계값은 `rule_id`와 버전으로 저장한다.
- 0건, 결측값, 표본 부족 상황에서는 증가율을 임의 계산하지 않고 `데이터 부족` 상태와 사용한 표본 수를 표시한다.

### 2.4 사전 정의 질의 10개

1. 최근 부정 VOC가 가장 많이 증가한 시설은 어디인가?
2. 조식 대기 관련 부정 리뷰가 증가한 기간의 운영지표를 보여줘.
3. 객실 청결 불만의 전주 대비 변화를 보여줘.
4. 주차 불만이 증가한 시간대와 관련 운영지표를 보여줘.
5. 방문객은 증가했지만 근무 인원이 유지된 시간대를 찾아줘.
6. 평균 대기시간과 부정 VOC가 함께 증가한 시설을 보여줘.
7. 특정 이슈의 근거 리뷰 원문과 근거 문장을 보여줘.
8. 이번 주 경영진용 주요 위험 이슈를 요약해줘.
9. 개선 우선순위가 높은 시설과 판단 근거를 보여줘.
10. 데이터 부족 또는 분석 실패가 발생한 항목을 보여줘.

각 질의의 공통 출력은 분석 기간, 시설, 주제, VOC 수치, 관련 운영지표, 원인 후보, 근거 리뷰 ID, 표본 수, 적용 규칙·모델 버전과 한계 문구다.

### 2.5 평가·성능·추적 기준

- 감성·주제 평가는 최소 150개 리뷰로 구성한 고정 평가셋을 사용한다. 라벨링 지침을 문서화하고 평가셋의 20%는 2명이 교차 검수해 불일치를 합의한다.
- 감성과 주제 분류는 각각 Macro F1 0.75 이상을 목표로 하며 주제별 표본 수, 오분류 사례와 한계를 함께 기록한다.
- 사전 정의 질의는 위 10개를 고정 테스트셋으로 사용하고 의도·기간·시설·지표·필수 출력이 모두 맞은 경우에만 성공으로 판정한다. 10개 중 8개 이상 통과해야 한다.
- 대시보드 성능은 VOC 10,000건과 30일치 운영 데이터에서 첫 실행을 제외한 10회 중 95%가 5초 이내여야 한다. 측정 장비, Python 버전, 캐시 상태를 결과에 기록한다.
- Agent 응답시간은 LLM API 호출을 포함해 10개 질의 중 90%가 30초 이내여야 하며 시간 초과 시 부분 결과와 재시도 안내를 제공한다.
- 각 Core MVP 요구사항은 구현 모듈, 화면, 테스트 케이스와 WBS 작업 ID 중 해당 항목을 연결해 추적한다.

## 3. 상세 요구사항

### 3.1 업무 요구사항

#### BIZ-001. VOC·운영 통합 분석

- **업무 영역:** 분석 업무 › VOC·운영 통합 분석
- **요구사항:** 워커힐 고객경험 담당자는 VOC 변화와 관련 운영지표 변화를 동일 시설·동일 기간 기준으로 함께 분석할 수 있어야 한다.
- **세부 설명:** 대상: 고객경험·운영 담당자. 출력: VOC 증감률, 운영지표 증감률, 비교 기간.
- **수용 기준:** Given 두 기간의 데이터가 있을 때, When 시설·주제를 선택하면, Then 두 기간의 지표와 증감률이 함께 표시된다.
- **우선순위·범위:** Must · MVP

#### BIZ-002. 원인 후보 표현

- **업무 영역:** 의사결정 › 원인 후보 표현
- **요구사항:** 시스템은 분석 결과를 확정 원인이 아닌 원인 후보·연관 지표·우선 점검 항목으로 제공해야 한다.
- **세부 설명:** 업무 규칙: 인과 표현 금지, 최종 판단은 운영자 수행.
- **수용 기준:** Given 교차분석 결과가 있을 때, When 설명을 생성하면, Then `원인 확정` 표현 없이 근거 수치와 한계가 표시된다.
- **우선순위·범위:** Must · MVP

#### BIZ-003. 역할별 정보 제공

- **업무 영역:** 사용자 › 역할별 정보 제공
- **요구사항:** 총지배인·경영진과 부서 담당자는 역할에 맞는 요약 수준과 데이터 범위를 조회할 수 있어야 한다.
- **세부 설명:** 경영진: 주간 요약·위험·우선순위. 실무자: 원문·시간대·시설 상세.
- **수용 기준:** Given 서로 다른 역할의 계정이 있을 때, When 같은 이슈를 조회하면, Then 권한별 화면과 상세 범위가 구분된다.
- **우선순위·범위:** Should · Enhanced MVP · 고객사 협의 필요

#### BIZ-004. 자동 실행 제외

- **업무 영역:** 범위 › 자동 실행 제외
- **요구사항:** 시스템은 대응 옵션을 제안하되 고객 응대·보상·인력 배치·운영 명령을 자동 실행하지 않아야 한다.
- **세부 설명:** 출력은 검토·선택·메모까지만 제공한다.
- **수용 기준:** Given 대응 옵션이 생성됐을 때, When 사용자가 선택하면, Then 검토 상태만 저장되고 외부 운영 시스템 명령은 발생하지 않는다.
- **우선순위·범위:** Must · MVP

### 3.2 기능 요구사항

#### FUN-001. 사내 사용자 로그인

- **업무 영역:** 인증 › 사내 사용자 로그인
- **요구사항:** Enhanced MVP에서는 데모 계정으로 로그인할 수 있어야 하며 실제 사내 인증 연동은 차기 범위로 둔다.
- **세부 설명:** 입력: 사전 등록한 데모 계정. 예외: 인증 실패 안내. 실제 SSO·사내 계정 연동은 고객사 협의 필요.
- **수용 기준:** Given 유효·무효 데모 계정이 있을 때, When 로그인하면, Then 유효 계정만 접속되고 실패 사유가 노출된다.
- **우선순위·범위:** Should · Enhanced MVP

#### FUN-002. 역할 기반 접근제어

- **업무 영역:** 권한 › 역할 기반 접근제어
- **요구사항:** Enhanced MVP에서는 경영진, 실무자, 관리자 3개 데모 역할별 메뉴와 데이터 접근을 구분해야 한다.
- **세부 설명:** 선행: FUN-001.
- **수용 기준:** Given 일반 담당자 계정일 때, When 관리자 URL에 접근하면, Then 접근이 거부되고 보안 로그가 남는다.
- **우선순위·범위:** Should · Enhanced MVP

#### FUN-003. VOC 파일 업로드

- **업무 영역:** 데이터 업로드 › VOC 파일 업로드
- **요구사항:** 관리자는 고객 리뷰·홈페이지 VOC·만족도 조사 CSV 파일을 업로드할 수 있어야 한다.
- **세부 설명:** 입력: CSV. 출력: 처리 건수·성공·오류 상태.
- **수용 기준:** Given 정상 CSV일 때, When 업로드하면, Then 파일과 처리 상태가 등록된다. 실제 채널별 형식은 실제 데이터 확인 필요.
- **우선순위·범위:** Must · MVP

#### FUN-004. 운영 데이터 업로드

- **업무 영역:** 데이터 업로드 › 운영 데이터 업로드
- **요구사항:** 관리자는 시설별 방문객·인력·대기시간·수용률·객실 점유율 운영 데이터를 업로드할 수 있어야 한다.
- **세부 설명:** 입력: 프로젝트용 합성 CSV.
- **수용 기준:** Given 필수 컬럼을 가진 파일일 때, When 업로드하면, Then 시설·일자·시간 기준 레코드가 적재된다.
- **우선순위·범위:** Must · MVP · 합성 데이터

#### FUN-005. 업로드 상태·오류 제공

- **업무 영역:** 데이터 업로드 › 업로드 상태·오류 제공

- **세부 설명:** 실제 보존 기간과 삭제 승인 절차는 고객사 협의 필요.
- **수용 기준:** Given 보존 정책 초안이 있을 때, When 데이터 유형별 정책을 검토하면, Then 원본·정제본·분석 결과·로그의 보존 기간, 삭제 승인자와 삭제 방법이 문서화된다.
- **우선순위·범위:** Should · Enhanced MVP · 고객사 협의 필요

### 3.11 검증·테스트 요구사항

#### TST-001. 데이터 품질 테스트

- **업무 영역:** 데이터 › 데이터 품질 테스트
- **요구사항:** 필수 컬럼 누락, 중복 ID, 잘못된 날짜, 결측값과 개인정보 마스킹 시나리오를 자동 검증해야 한다.
- **수용 기준:** Given 필수 컬럼 누락·중복 ID·잘못된 날짜·결측값·마스킹 대상이 포함된 오류 샘플과 정상 샘플이 있을 때, When 적재 테스트를 실행하면, Then 오류 샘플은 100% 탐지되고 정상 샘플은 오차 없이 적재된다.
- **우선순위·범위:** Must · MVP

#### TST-002. 분류 성능 평가

- **업무 영역:** AI › 분류 성능 평가
- **요구사항:** 감성·주제 분류는 고정 라벨 데이터셋으로 Macro F1과 오류 사례를 평가해야 한다.
- **수용 기준:** 최소 150개 리뷰의 고정 평가셋에서 감성·주제별 Macro F1 0.75 이상을 확인하고 주제별 표본 수, 오분류 사례와 한계를 기록한다. 평가셋 20%는 2명이 교차 검수한다.
- **우선순위·범위:** Must · MVP

#### TST-003. 지원 질의 평가

- **업무 영역:** Agent › 지원 질의 평가
- **요구사항:** 지원 질문 10개의 의도·기간·시설 추출과 필수 결과 항목을 검증해야 한다.
- **수용 기준:** Given 2.4절의 고정 질의 10개일 때, When Agent 테스트를 실행하면, Then 의도·기간·시설·지표·필수 출력이 모두 맞은 질의가 8개 이상이고 실패 케이스와 제한사항이 기록된다.
- **우선순위·범위:** Must · MVP

#### TST-004. 역할별 접근 테스트

- **업무 영역:** 권한 › 역할별 접근 테스트
- **요구사항:** 각 역할의 허용·차단 메뉴, API, 다운로드를 테스트해야 한다.
- **수용 기준:** Given 경영진·실무자·관리자 데모 계정일 때, When 역할별 메뉴·API·다운로드를 요청하면, Then 비인가 접근은 100% 차단되고 허용 기능은 정상 조회된다.
- **우선순위·범위:** Should · Enhanced MVP

#### TST-005. 장애·예외 테스트

- **업무 영역:** 장애 › 장애·예외 테스트
- **요구사항:** LLM 오류, 데이터 부족, 파일 오류, 처리 시간 초과 상황에서 부분 결과와 안내 메시지를 검증해야 한다.
- **수용 기준:** Given LLM 오류·데이터 부족·파일 오류·처리 시간 초과 시나리오일 때, When 각 오류를 발생시키면, Then 서비스 전체 중단 없이 부분 결과 또는 재시도 경로와 오류 로그가 제공된다.
- **우선순위·범위:** Must · MVP

### 3.12 차기 고도화 요구사항

## 5. 누락·충돌 점검 결과

- **기능 선후관계:** 인증·권한 → 업로드·검증 → 전처리·마스킹 → 분류·KPI → 교차분석·규칙 → Agent·화면·리포트 순서로 정의했다.
- **중복 요구사항:** 동일 목적 요구사항을 데이터 처리, 분석, 화면, 검증 관점으로 구분했으며 ID 중복은 없다.
- **검증 불가능 표현:** `편리한`, `좋은`, `고도화된` 같은 표현을 제거하고 Must 항목에 Given/When/Then 또는 정량 통과 기준을 포함했다.
- **MVP 범위 초과:** 실제 시스템 연동, PDF, 실시간 알림, 티켓, 예측, 다지점·다국어는 Could/Future로 분리했다.
- **실제 정보 오인 위험:** 실제 시설명·시스템 제품명·데이터 규모·인력 수를 만들지 않았고 미확정 항목을 명시했다.
- **데이터 의존 위험:** 실제 스키마, 성능 기준 데이터 규모, 라벨 평가셋, 임계값과 개인정보 패턴은 실제 데이터 확인 전 확정할 수 없다.
- **AI 역할 충돌:** 수치 계산은 결정론적 로직, LLM은 분류·질의 해석·설명 생성으로 분리했다.

## 7. 문서 활용 원칙

- 수치 계산은 SQL·Pandas·통계 규칙 등 결정론적 로직으로 수행합니다.
- LLM은 분류, 질의 해석, 설명 생성에 사용하며 상관관계를 인과관계로 단정하지 않습니다.
- 대응안은 운영자가 검토할 수 있는 선택지로 제공하며 고객 응대, 보상, 인력 배치 등을 자동 실행하지 않습니다.
- 실제 데이터 구조, 권한 체계, 연동 방식, 임계값은 고객사 확인 후 확정합니다.
